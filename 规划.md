## 五、下一步具体行动（按优先级）

### 1. 高优先级：在 exp_04 难样本集上重跑三个实验

exp_04 的 42 个样本已就位（含 16 个难样本），现在是验证 LLM / RAG / 传统工具在难样本上真实表现的时候。

具体做法：
1. 在 `experiments/exp_04_hard_samples/` 下复用 / 调整 `run_experiment.py`、`run_baseline.py`、`run_rag_experiment.py`。
2. 特别注意跨文件样本（`hard_crossfile_*_input.py` + `*_sink.py`）：需要把 input 文件作为上下文一起送入模型，否则 sink 文件会缺少污染源信息。
3. 生成 `experiments/exp_04_hard_samples/exp_04_report.md`，对比典型样本与难样本的召回率、误报率、耗时。

### 2. 高优先级：重复实验与置信区间（P1-4）

temperature=0.1 仍有随机性。为 `run_experiment.py` / `run_rag_experiment.py` 增加 `--repeat N`（默认 3），计算：
- 每样本 N 次中判定为漏洞的比例
- 多数表决后的整体指标
- 总体准确率的 95% 置信区间（Wilson score interval）
- 耗时的均值 / 标准差 / 中位数

### 3. 中优先级：RAG 消融对照实验（P1-5）

在难样本集上跑 4 组对照，排除"注入文本变长"或"随机注入"的干扰：
- A 组：RAG+LLM（当前实现）
- B 组：纯 LLM（无 RAG）
- C 组：随机知识注入（Top-K 条与样本无关）
- D 组：等长无关文本注入
只有 A 显著优于 B/C/D，才能严格证明 RAG 有用。

### 4. 中优先级：Prompt 工程对比

设计 3 套 Prompt 在同一批难样本上对比：
- 零样本（当前版本）
- Few-shot（带 2-3 个典型漏洞示例）
- 思维链（要求模型先逐步分析再下结论）

### 5. 中优先级：Bandit 告警级别过滤（P2-6）

当前 `run_baseline.py` 把 `B404 import_subprocess` 等信息性 finding 也计为漏洞，导致 safe_02 被记 3 条误报。需要：
- 解析每条 finding 的 `severity` 和 `confidence`
- 默认只计 `severity >= LOW` 且 `confidence >= MEDIUM` 的告警
- 提供 `--bandit-min-severity` 参数，报告中给出严格/宽松两种口径

### 6. 中优先级：RAG Top-K 对比（P2-8）

分别跑 K=1,3,5,10，记录检出率、误报率、Top-1 类型命中率、平均检索 distance、耗时，给出推荐 K 值。

### 7. 中优先级：AST 代码切片

用 tree-sitter 把长文件按函数切分，解决上下文窗口与注意力衰减，重点看 `hard_longfile_*` 样本的检出率变化。

### 8. 低优先级：系统 MVP 开发

待实验阶段结论稳定后再开始，避免需求反复推翻。

## 六、关键备忘

- 先不要搭前后端框架，需求会在难样本实验后明确。
- 所有实验过程和 Prompt 迭代都要保留，后续写论文直接可用。
- 模型名称注意核实：Ollama 中实际可用的模型名要与代码里配置的一致。
- 显存管理：每次实验脚本跑完必须主动从显存卸载模型（`keep_alive=0`）。
- 大模型文件、向量库数据绝不入 git。
- **盲测状态**：源码与 prompt 中已无答案泄露注释，但文件名（如 `sql_injection_01.py`、`typical_01_sql.py`）仍可能泄露漏洞类型；后续若要求严格盲测，可对文件名做中性化处理。
- **答辩核心故事线**：传统静态分析工具在 CI/CD 中表现优秀，但面对复杂业务逻辑、绕过式过滤、跨函数污点时力不从心；本系统利用本地 LLM + RAG 知识库实现语义级代码理解，在保证检出率的同时生成可执行修复建议，将人工审计时间从 30 分钟级降到 5 分钟级。
- **工程遗留**：包名反模式修正（`src` → `graduation_project`）和结果文件按时间戳命名越早做影响面越小，当前不阻塞运行。
