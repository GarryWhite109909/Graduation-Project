# 训练方法体系

> 本文档梳理 LLM 微调训练方法体系，从"为什么效果差"到"如何做知识注入"，给本项目（exp\_06\_finetune 漏洞检测微调）提供方法论支撑。
> 配套实施细节见 [改进.md](file:///home/zane/文档/code/毕业设计/docs/改进.md)。

***

## 一、背景：LLM 微调与视觉模型微调的本质差异

本项目先前训练 YOLO 模型效果显著，但 LLM（Qwen2.5-Coder）LoRA 微调出现 FPR 飙升、missing feature 类漏洞回归等问题。根因不是技术不熟练，而是两类任务的性质根本不同：

| 维度     | YOLO 目标检测         | LLM 漏洞检测                     |
| :----- | :---------------- | :--------------------------- |
| 任务性质   | 视觉感知（找边界框）        | 代码语义推理（追踪数据流/控制流/缺失检查）       |
| 类别相似度  | 狗/猫/车共享大量视觉特征     | SQL注入 vs CSRF vs 反序列化，模式完全不同 |
| 单样本信息量 | 一张图多个目标框          | 一段代码一个标签                     |
| 微调方式   | 全参数微调（除 backbone） | LoRA r=8 只更新约 0.1% 参数        |
| 迁移学习起点 | COCO 预训练已具备检测能力   | Qwen-Coder 懂代码但不懂"漏洞推理"这个新技能 |
| 数据规模   | COCO 80 类 × 数十万图  | 823 样本覆盖 15+ 漏洞类型            |

**核心结论**：LoRA 的本质是"风格微调"，不是"知识注入"。低 rank 的 LoRA（r=8）只能调整输出格式与风格，无法让模型学会新的推理能力。漏洞检测需要的是**新推理模式**，这超出了 r=8 LoRA 的容量。

***

## 二、核心概念：风格微调 vs 知识注入

判断训练目标是哪种，决定方法选择：

| 类型       | 目标                | 典型场景           | 所需容量        |
| :------- | :---------------- | :------------- | :---------- |
| **风格微调** | 调整输出格式、语气、回答风格    | JSON 格式化、角色扮演  | LoRA r=8 足够 |
| **知识注入** | 学会新概念、新推理模式、新领域知识 | 漏洞检测、医学问答、法律推理 | r=32+ 或全参数  |

### 数据表达的关键区分

知识注入的成败不只看方法，更看**数据的表达方式**：

```
标签→反推理由（错误做法）：
  "已知是 SQL 注入，所以解释为什么" → 模型学到"模式→标签"映射

推理→得出标签（正确做法）：
  分析污染源 → 追踪数据流 → 发现无防御 → 得出 SQL 注入
  → 模型学到推理链
```

本项目 [train\_chatml\_v2.jsonl](file:///home/zane/文档/code/毕业设计/experiments/exp_06_finetune/data/train_chatml_v2.jsonl) 已包含 CoT 推理过程（污染源→sink→数据流→防御检查→结论），数据表达正确，但容量配置不足。

## 三、知识注入的 5 个层次

按效果递增排列。每层标注对本项目的适用性。

### L1：RAG（检索增强，不训练）

**原理**：把领域知识做成向量库，推理时 retrieve 进 context，让模型有"参照标准"。

**对本项目**：把 CWE 规则、OWASP Top 10、训练集漏洞示例做成向量库，推理时 retrieve top-3 相关规则进 prompt。

```text
[用户代码] + [检索到的 CWE-89 SQL 注入规则] + [同类漏洞示例]
              ↓
          模型判断
```

**优点**：100% 准确、可热更新、不需训练
**缺点**：受 context length 限制、依赖检索质量、推理变慢
**预期效果**：FPR 立即下降，因为模型有判断参照

### L2：Continued Pretraining（继续预训练）

**原理**：在 SFT 之前，先在大量无标签领域语料上做 causal LM 继续预训练，注入"领域语言模式"。

**数据要求**：

- CVE 数据库描述
- CWE 规则详情
- OWASP Top 10 文档
- 漏洞分析博客/报告
- 规模：10–100MB 纯文本

**训练配置**：

```python
# 全参数或 r=128 LoRA
TrainingArguments(
    learning_rate=2e-5,
    num_train_epochs=1,  # 继续预训练 1-2 epoch 即可
    per_device_train_batch_size=1,
    gradient_accumulation_steps=16,
)
# loss = causal LM (next token prediction)，不是 SFT 的 assistant_only_loss
```

**对本项目**：让模型从"懂代码"升级为"懂漏洞术语"，建立漏洞领域语言先验，再 SFT 时收敛更快。
**优点**：真正的"知识注入"，效果持久
**缺点**：需要收集领域语料，计算成本较高

### L3：高容量 LoRA + CoT 蒸馏

<br />

**配置对比**：

| 参数               | 当前配置     | 推荐配置（2026-07-18 修正）                 | 理由                             |
| :--------------- | :------- | :------------------------------------ | :----------------------------- |
| `lora_r`         | 8        | 32（保守）/ 64（激进）                    | r=8 在弱基座或复杂任务上容量不足；7B 90.8% 强基座上 r=8 也可轻量注入 |
| `lora_alpha`     | 16       | 64 / 128                              | 保持 alpha=2\*r                  |
| `lora_dropout`   | 0.1      | 0.05~0.1                              | 小数据保持 0.1 防过拟合                 |
| `epochs`         | 1        | 1~2（由 early stopping 决定）             | 7B 基座已很强，多 epoch 易过拟合          |
| `learning_rate`  | 1e-5     | 5e-5 / 1e-4（先 sweep，再决定是否上 2e-4）   | lr 调优比方法选择更重要；直接 2e-4 可能过拟合  |
| `use_rslora`     | False    | True                                  | 零成本提升，高 rank 更稳定                |
| `use_dora`       | False    | True（如 4bit+ROCm 验证通过）              | 结构化变体，RLVR/SFT 均优于标准 LoRA      |
| `target_modules` | 7 个 proj | 7 个 proj（足够）                        | 7B 强基座不必加 embed/lm_head          |

**关键 CoT 重蒸馏**（[generate\_distill\_data.py](file:///home/zane/文档/code/毕业设计/experiments/exp_06_finetune/scripts/generate_distill_data.py) 已就绪）：

```text
旧 CoT（标签→反推）：分析污染源是 request.args，所以是 SQL 注入

新 CoT（推理→标签）：
  Step1: 这段代码功能是搜索产品
  Step2: 用户输入从 request.args.get('q') 进入
  Step3: 追踪 keyword 变量，未经过滤
  Step4: keyword 通过字符串拼接进入 SQL 语句
  Step5: cursor.execute 直接执行拼接后的 SQL
  Step6: 检查防御措施：无参数化查询、无 ORM、无输入校验
  Step7: 综合判断：用户可控输入直达 SQL sink，且无防御 → SQL 注入
```

每一步推理基于前一步的结论，不是从最终标签倒推。

### L4：全参数微调（16GB 显存需特殊优化）

**显存核算**（修正先前错误判断）：

全参数微调显存 ≈ 12 × 模型参数量(GB) + 激活值，分解如下：

| 组成                   | 3B 模型 fp16  | 7B 模型 fp16   |
| :------------------- | :---------- | :----------- |
| 模型参数                 | 6 GB        | 14 GB        |
| 梯度                   | 6 GB        | 14 GB        |
| AdamW 状态（fp32 的 m+v） | 48 GB       | 112 GB       |
| 激活值（梯度检查点后）          | \~2 GB      | \~4 GB       |
| **总计**               | **\~62 GB** | **\~144 GB** |

**结论**：标准 AdamW + 3B 全参数微调在 16GB 显卡上 **必然 OOM**。必须用以下优化技术才能勉强可行：

| 优化方案                              | 3B 显存估算 | 7B 显存估算 | 备注            |
| :-------------------------------- | :------ | :------ | :------------ |
| 标准 AdamW                          | 62 GB   | 144 GB  | ❌ 16GB 不可行    |
| 8bit AdamW（paged\_adamw\_8bit）    | \~20 GB | \~48 GB | ❌ 仍不够         |
| **GaLore**（梯度低秩投影）                | \~10 GB | \~18 GB | ✅ 3B 可行，7B 仍险 |
| **Adafactor**（无状态优化器）             | \~14 GB | \~32 GB | ✅ 3B 可行       |
| **8bit Adam + CPU offload 优化器状态** | \~10 GB | \~18 GB | ✅ 慢但可行        |

**推荐配置（3B + GaLore，16GB 可行）**：

```python
# GaLore：投影梯度到低秩空间，全参数训练但显存接近 LoRA
from gallora import GaLoreOptim
TrainingArguments(
    optim="galore_adamw",         # GaLore 优化器
    optim_args={"rank": 128},     # 投影 rank
    learning_rate=2e-5,
    num_train_epochs=3,
    per_device_train_batch_size=1,
    gradient_accumulation_steps=16,
    gradient_checkpointing=True,
)
# 关键：混入 80% 漏洞数据 + 20% 通用代码数据，防遗忘
```

**对本项目**：

1. 3B + GaLore 在 16GB 上理论可行，但 ROCm 上 GaLore 兼容性需验证
2. 7B 全参数微调在 16GB 上 **完全不可行**，仍需 QLoRA
3. 全参数微调收益相比 r=64 高容量 LoRA 提升有限，性价比不高

### L5：多阶段组合训练（业界 SOTA，效果最好）

ChatGPT、Llama-3 等采用的标准 pipeline。对本项目分三阶段：

```text
Stage 1: Continued Pretraining（1-2 epochs）
  ├─ 数据: CVE/CWE/OWASP 领域语料
  ├─ 方法: causal LM loss, 全参数或 r=128 LoRA
  └─ 目标: 注入"漏洞领域知识"

Stage 2: SFT with CoT (3 epochs)
  ├─ 数据: train_chatml_v2.jsonl 的 823 条 CoT 样本
  ├─ 方法: r=64 LoRA, assistant_only_loss=True
  ├─ 脚本: train_qlora.py（需调参）
  └─ 目标: 注入"推理技能"

Stage 3: DPO 对比学习 (1-2 epochs)
  ├─ 数据: dpo_preference_pairs_v3.jsonl（98 条）
  ├─ 方法: DPO loss
  ├─ 脚本: train_dpo.py（已就绪）
  └─ 目标: 校准判断边界，降 FPR
```

三阶段递进：知识 → 技能 → 校准。

## 四、类人学习范式：预习-上课-复习

借鉴人类学习的"预习→上课→复习"三阶段范式（达摩院 SaCLog，ACL 2021；UCLA LFR，arXiv:2409.06131），用于训练数据策略优化。

### 三阶段对本项目的适用性

| 阶段     | 含义                             | 对本项目      | 实施                                    |
| :----- | :----------------------------- | :-------- | :------------------------------------ |
| **预习** | 先看任务 schema/概念结构建立先验           | ❌ 用处有限    | Qwen-Coder 已预训练代码，"预习"在预训练阶段完成        |
| **上课** | 课程学习（Curriculum Learning），由易到难 | ✅ 适用      | 先训明显的 SQL/XSS 注入，再训 CSRF/authz 等缺失型漏洞 |
| **复习** | 错题增强：找出错样本 → 数据增强 → 重训         | ✅ **最适用** | 直接修复本项目 missing feature 类漏洞的回归问题      |

### 复习机制的闭环实现

针对本项目 3B ft 出现的 missing feature 回归（CSRF/authz/session/integer\_overflow 从 TP→FN）：

1. **第一轮训练**完成，在 dev/test 集上找出错样本（CSRF、authz 等 missing feature 类）
2. **数据增强**：针对错题生成同类变体（[supplement\_hard\_samples.py](file:///home/zane/文档/code/毕业设计/experiments/exp_06_finetune/scripts/supplement_hard_samples.py) 已实现类似机制）
3. **加权采样**：第二轮训练时对"易错样本"加权
4. **再训练**：在新数据上继续训练（用 `--resume` 从 best checkpoint 接续）

本项目已有 `supplement_*.py` 系列脚本（共 14 个，覆盖 SSTI/SpEL/NoSQL/CWE 归因/长文件防御等），但未串成"训练-复习-再训练"的闭环。

### 课程学习的实施建议

按漏洞类型的"可学习难度"分阶段：

```text
Easy Stage（特征明显的注入类）：
  SQL 注入、XSS、命令注入、路径穿越
  → 训练数据充足、模式清晰

Medium Stage（语义判断类）：
  反序列化、SSTI、XXE、弱密码学、弱随机数
  → 需理解 API 语义

Hard Stage（缺失型漏洞，本项目弱点）：
  CSRF、缺失认证/授权、session 管理、integer_overflow
  → 需"反证"推理（应该有的检查缺失）
```

每阶段训练 1 epoch，逐步加难。

## 五、本项目推荐的多阶段训练 Pipeline

按性价比排序的执行路径：

### 推荐路径 A：低成本验证（先做）

1. **L3 改进**：修改 [train\_qlora.py](file:///home/zane/文档/code/毕业设计/experiments/exp_06_finetune/scripts/train_qlora.py) 配置 `r=8→r=64, alpha=16→128, epochs=1→3, lr=1e-5→2e-4`，跑一版看 FPR 是否从 42.3% 降下来
2. **L1 RAG**：把 CWE 规则做成向量库，evaluate 时检索进 prompt（不需训练，立竿见影）

### 推荐路径 B：完整多阶段训练（效果最好）

按 L5 三阶段 pipeline：

| 阶段           | 数据                               | 方法                    | 脚本                                                                                                   | 目标   |
| :----------- | :------------------------------- | :-------------------- | :--------------------------------------------------------------------------------------------------- | :--- |
| Stage 1: CPT | CVE/CWE/OWASP 语料                 | causal LM, r=128 LoRA | 待建                                                                                                   | 知识注入 |
| Stage 2: SFT | train\_chatml\_v2.jsonl          | r=64 LoRA, 3 epochs   | [train\_qlora.py](file:///home/zane/文档/code/毕业设计/experiments/exp_06_finetune/scripts/train_qlora.py) | 推理技能 |
| Stage 3: DPO | dpo\_preference\_pairs\_v3.jsonl | DPO loss, 1-2 epochs  | [train\_dpo.py](file:///home/zane/文档/code/毕业设计/experiments/exp_06_finetune/scripts/train_dpo.py)     | 边界校准 |

### 推荐路径 C：数据闭环（修复 missing feature 回归）

按"复习机制"做 hard sample mining 闭环：

1. 用当前 best 模型跑 eval，找出错的 missing feature 样本
2. 用 Qwen3-30B teacher 重新生成这些样本的 CoT
3. 用 [augment\_data.py](file:///home/zane/文档/code/毕业设计/experiments/exp_06_finetune/scripts/augment_data.py) 做数据增强
4. 合并到训练集，第二轮训练用加权采样
5. 评估 → 找新错题 → 再增强 → 再训练（迭代 2-3 轮）

## 六、关键配置参数参考

### LoRA 配置对比

| 场景             | r   | alpha | dropout | target\_modules | lr          | epochs | rsLoRA | DoRA |
| :------------- | :-- | :---- | :------ | :-------------- | :---------- | :----- | :----- | :--- |
| 风格微调（JSON 格式化） | 8   | 16    | 0.1     | 7 个 proj        | 1e-5        | 1      | 可选     | 否    |
| 技能注入（本项目 Phase 1）| 8   | 16    | 0.1     | 7 个 proj        | 5e-5 / 1e-4 | 1      | ✅      | 探测中 |
| 技能注入（本项目 Phase 2）| 32  | 64    | 0.1     | 7 个 proj        | Phase 1 最优  | 1-2    | ✅      | 如通过 |
| 知识注入（CPT 阶段）   | 64  | 128   | 0.05    | 7 proj + embed + lm\_head | 2e-5 | 1-2    | ✅      | 可选   |

> **Phase 1 实测 dev_loss 排行榜**（7B, r=8, e=1）：
> | 配置 | dev_loss | 相对 lr=1e-5 baseline |
> |---|---|---|
> | lr=1e-5 baseline | 1.4362 | — |
> | lr=5e-5 | 1.1100 | -22.7% |
> | lr=1e-4 | 0.9806 | -31.7% |
> | lr=5e-5 + rsLoRA | 1.0100 | -29.7% |
> | lr=1e-4 + rsLoRA | **0.8892** | **-38.1%** |
>
> 说明 lr 提升贡献最大，rsLoRA 在同 lr 下进一步降 loss。DoRA+rsLoRA 在 ROCm 7.2+4bit 上已跑通（run 5 进行中）。

| 任务         | 数据量           | 备注               |

## 八、2025-2026 最新 PEFT 方法借鉴

通过网络检索梳理的近期（2025-2026）发表的相关方法。这些方法在 LoRA 基础上做了改进，或提供了新的知识注入思路，可作为本项目后续实验的备选方案。

### 8.1 LoRA 改进型（替代标准 LoRA，零/低成本升级）

| 方法          | 核心思想                                            | 优势                             | 论文/来源                               | 对本项目适用性                           |
| :---------- | :---------------------------------------------- | :----------------------------- | :---------------------------------- | :-------------------------------- |
| **DoRA**    | 把权重分解为 magnitude + direction，LoRA 只更新 direction | 在多项任务上优于 LoRA，RLVR 场景甚至超过全参数微调 | Liu et al. 2024, ICLR 2024          | ✅ 推荐，PEFT 库已支持                    |
| **rsLoRA**  | 缩放因子从 `1/r` 改为 `1/√r`                           | 高 rank 时训练更稳定，零额外成本            | Hayou et al. 2024                   | ✅ 强烈推荐，改 `use_rslora=True` 即可     |
| **AdaLoRA** | 动态分配 rank 预算到不同层                                | 根据重要性自动调参                      | Zhang et al. 2023                   | ⚠️ 多语言任务表现稳定，推理任务收益一般             |
| **PiSSA**   | SVD 分解初始化 LoRA，只训主成分                            | 初始对齐预训练权重方向                    | Meng et al. 2024                    | ⚠️ RLVR 场景发现 spectral collapse，慎用 |
| **SORA**    | 正交旋转 + 对角 conditioning                          | 比 DoRA 更高效且效果更好                | Chang et al. 2025, arXiv:2511.00051 | ⚠️ 较新，工程验证少                       |

**关键实证发现**（[arXiv:2602.04998](https://www.arxiv.org/pdf/2602.04998), 2026）：

> 不同 LoRA 变体在**学习率充分调优后**性能差距仅 1-2%。也就是说，方法选择的重要性不如**学习率调优**。本项目当前 lr=1e-5 太低，应优先调 lr 而非换方法。

**对本项目的建议**：优先采用 **DoRA + rsLoRA**（PEFT 库直接支持，零额外成本），但 lr 不要直接跳到 2e-4，应先做 sweep：

```python
from peft import LoraConfig
LoraConfig(
    r=32,                   # 7B 强基座用 r=32 更稳，r=64 作为激进备选
    lora_alpha=64,
    lora_dropout=0.1,
    use_rslora=True,        # 启用 rsLoRA，缩放因子改 1/√r
    use_dora=True,          # 启用 DoRA，权重分解为 magnitude+direction
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj",
                    "gate_proj", "up_proj", "down_proj"],
)
```

> **lr 优先级高于方法选择**（2026-07-18 更新）：
> - [arXiv:2602.04998](https://www.arxiv.org/pdf/2602.04998) 指出 LoRA 变体在学习率调优后差距仅 1-2%。
> - 本项目 Phase 1 实测：lr 从 1e-5 提到 1e-4，dev_loss 从 1.4362 降到 0.9806（-31.7%），同配置下加 rsLoRA 再降到 0.8892（-9.3%）。
> - 因此**先调 lr，再加 rsLoRA/DoRA**。直接 r=64+e=3+lr=2e-4 三重叠加对 7B 90.8% 强基座过重，易过拟合。

### 8.2 显存优化型（解决 16GB 全参数微调 OOM）

| 方法                         | 核心思想                          | 显存节省          | 对本项目                      |
| :------------------------- | :---------------------------- | :------------ | :------------------------ |
| **GaLore**                 | 把梯度投影到低秩空间再优化，全参数训练但显存接近 LoRA | 3B 降至 \~10GB  | ✅ 16GB 可跑 3B 全参数          |
| **Adafactor**              | 无状态优化器（不存 m/v）                | 3B 降至 \~14GB  | ✅ 但收敛慢                    |
| **LLMQ**                   | FP8 + 优化器 offload + 8bit 训练   | 16GB 跑 7B 预训练 | ❌ 仅支持 NVIDIA，AMD/ROCm 不支持 |
| **DeepSpeed ZeRO-offload** | 优化器状态 offload 到 CPU           | 显存换速度         | ⚠️ ROCm 兼容性需验证            |

**GaLore 关键引用**：[arXiv:2403.07404](https://arxiv.org/abs/2403.07404)（Zhao et al. 2024），论文证明能在 24GB 显存上跑 7B 全参数微调，3B 模型在 16GB 上理论可行。

### 8.3 知识注入型（针对 L2 Continued Pretraining 的改进）

#### KnItLM（ICLR 2026 投稿，[OpenReview](https://openreview.net/forum?id=2uctT30vTS)）

**核心思想**：不在 Instruct 模型上做 CPT（会破坏 instruction following），而是在 **base 模型** 上做 CPT with LoRA 注入知识，然后把 LoRA 权重合并到 Instruct 模型上。

**对本项目**：完美适配 L2 CPT 阶段的痛点——直接在 Qwen2.5-Coder-Instruct 上做 CPT 会破坏对话能力，KnItLM 流程是：

```text
1. 加载 Qwen2.5-Coder-7B（base，非 Instruct）
2. 在 CVE/CWE 语料上做 CPT with LoRA（注入知识）
3. 把 LoRA 权重 merge 到 Qwen2.5-Coder-7B-Instruct
4. 得到既懂漏洞知识又不丢对话能力的模型
```

效果：KnItLM 在 RAG 失败场景上把准确率从 54.17% 提升到 79.26%，且不需要后续 instruction tuning。

#### Prompt Distillation（TMLR 2025, [OpenReview](https://openreview.net/forum?id=drYpdSnRJk)）

**核心思想**：自蒸馏——用同一个 LLM 生成"带 context 的 QA pairs"作为 teacher，再训练学生模型（同 base + LoRA）模仿 teacher 的 token 分布（logits），而不只是 hard label。

**对本项目**：

- 不需要更大的 teacher 模型（Qwen3-30B 也省了）
- 比 SFT 更好，因为学的是 token 分布而非硬标签
- 甚至超过 RAG

**流程**：

```text
1. Teacher：Qwen2.5-Coder-3B + 漏洞规则 context → 生成回答（logits）
2. Student：Qwen2.5-Coder-3B + LoRA，无 context → 模仿 teacher 的 logits
3. Loss：KL(teacher_logits || student_logits)，而非 cross-entropy
```

**实现成本**：只需修改 [train\_qlora.py](file:///home/zane/文档/code/毕业设计/experiments/exp_06_finetune/scripts/train_qlora.py) 的 loss 函数，不需要新框架。

#### Knowledge-Instruct（[arXiv:2504.05571](https://www.arxiv.org/pdf/2504.05571)）

**核心思想**：用 instructions 包装 raw documents 做 CPT，避免破坏对话能力。比 raw CPT 在多跳问答上效果更好。

### 8.4 RLVR 场景下的 PEFT 评估（[arXiv:2512.23165](https://arxiv.org/pdf/2512.23165v2), 2025-12）

系统评估 12 种 PEFT 方法在 DeepSeek-R1-Distill 上的表现，关键发现：

1. **结构化变体（DoRA/MiSS/AdaLoRA）一致优于标准 LoRA**
2. **SVD 初始化方法（PiSSA/MiLoRA）出现 spectral collapse**，在 RL 优化下失败
3. **极端参数压缩（VeRA/Rank-1）严重限制推理能力**

**对本项目**：本项目虽是 SFT 不是 RLVR，但漏洞检测本质是推理任务，结论可借鉴——**优先 DoRA，避开 PiSSA**。

#### Phase 4 两种 teacher 后端（2026-07-18 增补）

[precompute\_teacher\_logits.py](file:///home/zane/文档/code/毕业设计/experiments/exp_06_finetune/scripts/precompute_teacher_logits.py) 支持 `--backend` 切换两种 teacher 加载方式：

| 后端 | Teacher 模型 | 显存 | Top-K | 蒸馏性质 | 推荐场景 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **transformers**（默认） | Qwen2.5-Coder-7B/14B-Instruct（4bit） | 8-11 GB | 50 | teacher forcing（teacher 看完整 answer 算 logits） | 显存充裕，要 token 级精度 |
| **ollama** ⭐ | Qwen3-Coder-30B-A3B MoE | **0（teacher 不占训练显存）** | 20（API 上限） | on-policy（teacher 自己生成 answer + logprobs） | 想用强 teacher 但显存受限 |

**ollama 后端关键优势**：
1. 借助 llama.cpp `-cmoe`（routed expert 放 CPU），30B MoE 在 16GB 上跑 100+ TPS
2. Teacher 完全独立服务，precompute 完即可 unload（`keep_alive=300`，遵循项目约定）
3. On-policy 蒸馏（[verl OPD 文档](https://verl.readthedocs.io/en/latest/algo/opd.html)）减少 exposure bias，2026 SOTA 做法
4. 30B MoE 是真 teacher（比 7B student 强 10x，有实质知识差距）

**ollama 后端限制**：
1. `top_logprobs` API 上限 20（已实测覆盖 99%+ 概率质量）
2. teacher 是 on-policy 生成，token 边界与 student 训练数据可能不一致（用 student tokenizer 重新 encode，对齐率 >95%）
3. 需手动起 Ollama 服务并拉模型

**使用方式**：

```bash
# transformers 后端（默认，14B teacher 升级版）
HF_HUB_OFFLINE=1 /home/zane/miniconda3/envs/AI/bin/python \
    experiments/exp_06_finetune/scripts/precompute_teacher_logits.py \
    --backend transformers \
    --model Qwen/Qwen2.5-Coder-14B-Instruct \
    --top-k 50

# ollama 后端（推荐，30B MoE teacher）
# 前置：ollama serve & ollama pull qwen3-coder:30b
/home/zane/miniconda3/envs/AI/bin/python \
    experiments/exp_06_finetune/scripts/precompute_teacher_logits.py \
    --backend ollama \
    --ollama-model qwen3-coder:30b \
    --top-k 20
```

**为啥不用更大模型直接训 student**：训练侧没有像 llama.cpp `-cmoe` 那样成熟的 MoE offload 工具。Axolotl `quantize_moe_experts` 能把 30B QLoRA 压到 23GB（[官方文档](http://docs.axolotl.ai/docs/expert_quantization.html)），但明确写了 **CUDA only**。DeepSpeed ZeRO-Offload（AMD 官方支持）只 offload optimizer state，不 offload expert。所以 16GB 上训 30B MoE 不可行，但用 Ollama 做推理 + precompute 完全可行。

## 十一、审查修订记录（2026-07-17）

> 对本文档的方法论进行实证校验，结合 [compare\_4way\_summary.md](file:///home/zane/文档/code/毕业设计/experiments/exp_06_finetune/results/compare_4way_summary.md) 实测结果修正 3 处过度激进的结论。这些修正已纳入 §9 统一执行计划。

### 整体判断

框架有道理，5 层知识注入、风格 vs 知识区分、显存核算、复习闭环、DoRA/rsLoRA 升级方向均成立。但具体参数建议有 3 处需修正，§9 已采纳。

### 修正 1：「r=8 只能风格微调」过于绝对

原文第 21 行断言 r=8 无法学新技能。但 7B r=8 e=1 lr=1e-5 实测：

| 指标 | 7B base | 7B ft (r=8) | 变化 |
|---|---|---|---|
| 严格 recall（CWE 对） | 33.3% | 41.0% | **+7.7pp** |
| CWE 错标 | 35 | 32 | -3 |
| CoT-JSON 不一致 | 2 | 0 | -2 |
| 幻觉率 | 63.6% | 56.1% | -7.5pp |

严格 recall +7.7pp、CWE 错标 -3 是**真实技能提升**，不是风格调整。问题不在 r=8，而在**弱基座（3B）上 r=8 容量不足**；强基座（7B）上 r=8 已能做轻量技能注入。**§9 已采纳**：Phase 1 仍用 r=8 做基线验证，Phase 2 用 r=32（而非原推荐的 r=64）。

### 修正 2：lr 1e-5 → 2e-4 跳跃太激进

原文 P0 建议直接跳到 2e-4（20 倍）。但 [train\_qlora.py:15-16](file:///home/zane/文档/code/毕业设计/experiments/exp_06_finetune/scripts/train_qlora.py#L15-L16) 注释明确"强干预会负蒸馏覆盖预训练知识"，且 lr=1e-5 已证明有效。直接跳 2e-4 有灾难性遗忘风险。**应做 lr sweep**（1e-5 / 5e-5 / 1e-4）而非定值。**§9 已采纳**：Phase 1 网格用 5e-5 / 1e-4，必要时 Phase 2 再加 2e-4。

### 修正 3：r=64 + e=3 + lr=2e-4 组合对 7B 过重

7B 基座已 90.8%，r=64（8 倍容量）+ 3 epochs + 高 lr 三重叠加，过拟合和遗忘风险都高。**r=32 是更稳的中间值**，配合 early stopping。**§9 已采纳**：Phase 2 用 r=32 + e=2 + early stopping。

- 全通过 → Phase 1 数据有效，无需重跑
- 可疑信号 → 跑一组 `adamw_torch` 对比验证（~1.5h）

**✅ 2026-07-18 22:15 实测结果**：5 个 run 全部通过验证

|---|---|---|---|---|---|

##### 12.2.2 PyTorch TunableOp 离线调优（+15% 端到端，但实测成本高）

**原理**：自动从 rocBLAS/hipBLASLt 挑选最优 GEMM kernel。AMD 官方博客实测 gfx1100 +15%。PyTorch 2.6+ 支持离线调优（先 record 再 tune，不污染训练）。

**⚠️ 2026-07-18 实测发现（PyTorch 2.11.0+rocm7.2）**：

1. **环境变量坑**：仅设 `PYTORCH_TUNABLEOP_ENABLED=1` 不够，必须用 Python API 显式启用：
   - `tunable.enable(True)` — 总开关
   - `tunable.record_untuned_enable(True)` — 录制模式（关键！）
   - `tunable.tuning_enable(False)` — 关闭在线调优
   - `PYTORCH_TUNABLEOP_FILE_NAME` 环境变量**不生效**，必须用 `tunable.set_filename()`

2. **录制正确流程**（[tunableop_offline_tune.sh](file:///home/zane/文档/code/毕业设计/experiments/exp_06_finetune/scripts/tunableop_offline_tune.sh) 已修正）：
   ```bash
   # Step 1 (Recording, 5min): 跑 5 step 训练，记录所有 GEMM shape
   export PYTORCH_TUNABLEOP_ENABLED=1
   export PYTORCH_TUNABLEOP_TUNING=0
   export PYTORCH_TUNABLEOP_RECORD_UNTUNED=1  # 关键！默认不录制
   # train_qlora.py 里会调用 Python API 显式启用
   # 输出: configs/tunableop_untuned0.csv（在 cwd，atexit 时 flush）

   # Step 2 (Tuning, 实测 1104 GEMM × 30s = ~9h): 离线调优
   # ⚠️ 默认参数下太慢！需减少迭代次数：
   #   tunable.set_max_tuning_iterations(10)  # 默认 100
   #   tunable.set_max_tuning_duration(3)     # 默认 30ms
   # 减少后: 1104 GEMM × 3-5s = ~1-1.5h
   # 用 Python API: tunable.tune_gemm_in_file("untuned0.csv")

   # Step 3 (Deploy, 自动): run_phase2_sft.sh 已配置自动加载
   ```

3. **实测时间成本**：本项目 7B 4bit QLoRA 训练采集到 1104 个唯一 GEMM shape，默认调优参数下每 GEMM ~30s，全量调优需 ~9 小时（比 Phase 2 训练本身 6h 还长）。减少迭代到 10 iters / 3ms 后仍需 ~1.5h。

4. **本项目决策**：仅调优了 46 个 GEMM（部分结果，`tunableop_tuned_partial.csv`），其余用默认实现。Phase 2 实测 step time 31s（vs Phase 1 的 73-76s），主要加速来自 AOTRITON attention（~50% 提速），TunableOp 贡献 <5%。

**何时跑**：Phase 2 启动前跑一次。但若时间紧张，可只跑 AOTRITON（零成本、收益更大），跳过 TunableOp 全量调优。

### 12.4 性能实测（2026-07-18 更新）

| 阶段 | 优化前 | 优化后（实测） | 加速 |
|---|---|---|---|
| Phase 1 step time（baseline） | 73-76 s/step | - | baseline（仅 sdpa） |
| Phase 2 step time（AOTRITON + 部分 TunableOp） | 73-76 s/step | **31 s/step** | **-58%**（远超预期） |
| Phase 2 总耗时（r=32, e=2, 176 步） | ~3.5h（预期） | **~1.5h（实测）** | 节省 2h |
| Phase 3 CPT（r=64, e=1） | ~5h（预期） | ~2.5h（按 Phase 2 外推） | 节省 2.5h |

**关键发现**：AOTRITON attention 的提速（~58%）远超文档原预期（20-40%）。推测原因：
1. Phase 1 用的 sdpa 在 gfx1200 上回退到 eager attention（最慢路径）
2. AOTRITON 启用后切到 Triton Flash Attention 路径
3. 从最慢 → 最快的跨度比"中速 → 最快"大

**TunableOp 贡献分析**：仅 46/1104 GEMM 调优（4.2% 覆盖），对 step time 贡献 <5%。AOTRITON 是主要加速来源。
