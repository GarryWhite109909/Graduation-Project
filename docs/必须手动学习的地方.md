## 一、样本标注（Ground Truth）—— 模型评分的"标准答案" ✅ 已完成

### 为什么要手工做

你让 LLM 判断一段代码有没有漏洞，那你怎么知道它说得对不对？总不能问模型"你对吗？"。

所以必须**提前人工定好标准答案**。这是整个实验的裁判标准，自动标注会引入"循环论证"。

### 大厂规范做法

对每个测试样本建立统一表格，字段固定如下：

| 字段 | 含义 | 示例 |
|---|---|---|
| 样本 ID | 唯一编号 | `SQLI-001` |
| 文件路径 | 代码位置 | `samples/sql_injection/sqli_01.py` |
| 漏洞类型 | CWE 编号 | `CWE-89`（SQL 注入） |
| 风险等级 | 高中低 | `High` |
| 是否真实存在漏洞 | 是/否 | `Yes` |
| 触发条件 | 怎么才会被利用 | `username 参数未过滤直接拼接入 SQL` |
| 污染路径（Source→Sink） | 数据从哪进、到哪用 | `request.args['username'] → query = "SELECT ..." + username → cursor.execute(query)` |
| 修复思路 | 一句话说明 | `使用参数化查询` |

### 小白操作步骤

1. 新建 Excel 或 Markdown 表格，表头就按上面 8 列来。
2. 每看完一段代码，先把"它到底有没有漏洞"这个判断做准。
3. 拿不准的 CWE 编号去 [MITRE CWE 官网](https://cwe.mitre.org/) 查，不要凭感觉写。
4. 全部标完后，找同学交叉检查一遍（大厂叫"双人 review"）。

## 二、RAG 漏洞知识库内容编写 —— 给 LLM 配一本"好教材" ✅ 已完成

### 为什么要手工做

直接把 CWE 原文向量化入库是没用的。CWE 原文是给安全专家看的，不是给模型看的。

你需要把它**翻译成"对代码检测有用的形式"**，也就是模型看了就知道"这段代码该不该报"。

### 大厂规范做法

建议扩展到 **30–50 条知识**，覆盖 OWASP Top 10 每类 3–5 条。每条知识用统一模板：

```markdown
## 知识条目：SQL 注入（CWE-89）

### 1. 漏洞定义
攻击者把恶意 SQL 拼进用户输入，改变数据库查询语义。

### 2. 典型代码特征（模型重点关注）
- 用户输入直接拼接到 SQL 字符串中
- 使用了 `+`、`.format()`、`%` 格式化构造 SQL
- 没有使用参数化查询（parameterized query）

### 3. 危险函数 / Sink 点
- `cursor.execute(sql_string)`
- `raw()`、`raw_query()`
- `sqlite3.connect(...).execute(...)`

### 4. 安全写法示例
```python
cursor.execute("SELECT * FROM users WHERE username = ?", (username,))
```

### 5. 常见绕过陷阱
- 假过滤：`if not username.startswith("'")` 并不能防注入
- 转义不完全：只转义单引号，没处理 `--`、`OR 1=1`

### 6. 修复方案
使用 ORM 或预编译语句，永远不要拼接用户输入。
```

### 小白操作步骤

1. 先列清单：OWASP Top 10 有哪些（SQL 注入、XSS、不安全的反序列化等）。
2. 每个类型写 3–5 条知识。
3. 每条知识必须包含：**特征 + 危险函数 + 安全示例 + 常见坑 + 修复方案**。
4. 全部写完后，运行 RAG 检索测试，看检索出来的内容是否真的能帮模型更准。

### RAG 调参要点

| 参数 | 建议 | 判断标准 |
|---|---|---|
| Top-K | 从 3 开始试 | 太多会引入噪音 |
| 相似度阈值 | 太低注入无关知识，太高漏掉相关知识 | 以 RAG 版比纯 LLM 更准为准 |
| Chunk 粒度 | 一个漏洞类型 = 一条知识 | 当前粒度合适 |
| Embedding 模型 | `sentence-transformers` 默认即可 | 后期可换更好的 |

## 三、Prompt 工程 —— 教 LLM"怎么说话、怎么思考" 🔄 部分完成

### 为什么要手工做

同样的模型，提示词写得好不好，结果差很多。这部分必须反复迭代。

### 大厂规范做法

至少做三套提示词，跑对比实验：

| 方案 | 核心区别 | 适用场景 |
|---|---|---|
| 零样本（Zero-shot） | 只给任务描述 | 看模型基础能力 |
| Few-shot | 给 2–3 个示例 | 让模型按固定格式输出 |
| 思维链（CoT） | 让模型"一步一步想" | 复杂判断场景 |

### 推荐输出格式协议（全项目统一 schema，唯一标准）

> ⚠️ 本 schema 为全项目唯一标准，所有实验脚本（exp_01 / exp_02 / exp_03 / 后续）的 Prompt 与
> 结果解析必须对齐此 schema。代码定义见 `graduation_project/schema.py` 的 `VERDICT_SCHEMA` 常量与
> `parse_verdict()` 函数（`graduation_project/llm_client.py` 仅为向后兼容 re-export）。无漏洞时 source/sink 填 `"N/A"`，risk_level 填 `"None"`。

要求模型必须按这个 JSON 输出，方便后续自动统计：

```json
{
  "has_vulnerability": true,
  "vulnerability_type": "CWE-89",
  "risk_level": "High",
  "source": "request.args['username']",
  "sink": "cursor.execute(query)",
  "explanation": "username 被直接拼接到 SQL 语句中...",
  "fix_suggestion": "使用参数化查询..."
}
```

### 小白操作步骤

1. 先写一版"系统提示词"，告诉它角色和输出格式。
2. 加 2–3 个输入输出示例，这就是 Few-shot。
3. 在提示词里加一句"请一步一步分析"，这就是 CoT。
4. 同一批样本跑三遍，看哪个准。

## 四、难样本设计 —— 专门考倒模型 ✅ 已完成

### 为什么要手工做

自动生成的样本往往"假难"，比如把变量名乱改。真正的难样本是**看起来像安全、实际有漏洞，或者看起来像漏洞、实际安全**。

### 大厂规范做法

难样本分三类：

| 类型 | 说明 | 示例 |
|---|---|---|
| 假过滤 | 有检查，但检查无效 | `if "'" not in username: ... 然后还是拼接` |
| 跨文件/跨函数污点 | 污染源和危险点不在同一处 | 输入在 `views.py`，SQL 在 `db.py` |
| 真实 CVE 片段 | 从真实漏洞里截取的代码 | WebGoat、DVWA 里的代码 |

### 小白操作步骤

1. 去 GitHub 搜 `OWASP WebGoat`、`DVWA`、`VulnHub` 找真实漏洞代码。
2. 对每个难样本，记录"难点在哪"，答辩时这就是分析点。
3. 难样本数量建议占总样本的 20–30%，不要全用简单样本。

## 五、修复建议质量评分 —— 看模型给的"药方"灵不灵 ⏳ 待开始

### 为什么要手工做

模型说"用参数化查询"很容易，但真让它写一段修复代码，可能语法错误、可能没修到根上、可能引入新 bug。

### 大厂规范做法

评分维度固定为：

| 维度 | 满分 | 评分标准 |
|---|---|---|
| 语法正确性 | 2 | 修复代码能否通过 Python 编译 |
| 漏洞是否真正修复 | 3 | 原来的利用路径是否还存在 |
| 是否引入新问题 | 2 | 有没有破坏原有功能或引入新 bug |
| 可维护性 | 1 | 代码是否清晰、符合规范 |
| 解释是否匹配修复 | 2 | 解释和实际修复是否一致 |

### 小白操作步骤

1. 对每个样本，把 LLM 生成的修复代码复制下来。
2. 能运行的就运行一下，看报不报错。
3. 按上表打分，满分 10 分。
4. 最后算平均分，作为系统修复质量的量化指标。

## 六、实验结论分析 —— 论文里最值钱的部分 🔄 部分完成

### 为什么要手工做

脚本只能给你数字："RAG 版准确率 85%，纯 LLM 版 72%"。

但**为什么 RAG 对 SQL 注入提升大、对 XSS 提升小**，这个解释只能人写。

### 大厂规范做法

写结论时用"三段论"：

1. **现象**：RAG 版在 SQL 注入上提升 15%，在 XSS 上只提升 3%。
2. **原因**：SQL 注入的 Sink 点集中（`execute` 类函数）、特征明显，RAG 知识命中率高；XSS 涉及 HTML/JS/Python 多种上下文，特征分散。
3. **启示**：后续应针对 XSS 补充更多上下文相关的知识条目。

### 小白操作步骤

1. 把实验结果表贴出来。
2. 挑 2–3 个最显著的差异。
3. 每个差异按"现象→原因→启示"写一段。
4. 不要只说"效果好"，要说"为什么好"。
