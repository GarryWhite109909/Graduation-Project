一、核心创新点（论文骨架）
1. LLM 的"语义理解" vs 传统工具的"模式匹配"
表格
维度	传统工具（Bandit/Semgrep）	LLM（Gemma4/GPT-OSS）
检测方式	固定规则、正则匹配	代码语义理解、上下文感知
漏洞覆盖	已知漏洞模式	可发现变体、非典型漏洞
输出形式	漏洞类型 + 规则编号	自然语言解释 + 修复建议 + 修复代码
多语言	工具专属规则集	跨语言统一理解
误报控制	规则泛化能力差	上下文判断过滤/净化逻辑
核心卖点：传统工具是"模式匹配"，本系统是"语义理解"。
2. RAG 知识库增强（差异化优势）
表格
对比	纯 LLM	RAG + LLM
知识来源	模型预训练知识（可能过时）	实时检索 OWASP/CWE/CVE 知识库
检出准确率	基准	提升（待实验验证）
解释质量	通用描述	结合具体漏洞模式的精准解释
修复建议	通用方案	基于知识库的最佳实践
关键实验：纯 LLM vs RAG+LLM 的对比测试（exp_03）。
3. 难样本的差异化优势（核心论据）
表格
样本类型	传统工具	LLM
假过滤（如 replace("'", "")）	❌ 漏报	✅ 识别过滤不充分
跨函数污点追踪	❌ 不支持	✅ 追踪数据流
上下文依赖（沙箱内安全/外危险）	❌ 一刀切	✅ 判断上下文
多语言混合（Python调C扩展）	❌ 需多工具链	✅ 统一理解
0day/新型漏洞模式	❌ 无规则	⚠️ 可能识别
关键实验：设计 5 个难样本，证明 LLM 在复杂场景下不可替代。
二、性能与实用性的平衡论证
4. 速度 vs 质量的权衡
表格
指标	Bandit	Semgrep	LLM (Gemma4 26B)
单样本耗时	0.5s	2s	56s
人工理解时间	30分钟/漏洞	30分钟/漏洞	5分钟/漏洞
修复代码生成	❌	❌	✅
整体效率	低	低	高（省人工时间）
核心论点：LLM 慢 100 倍，但把人工审计时间从 30 分钟降到 5 分钟，整体效率提升。

核心论点：定位为辅助审计工具，不是替代，是增强；面向具备 GPU 资源的团队。

价值：不是单一工具，是端到端的智能审计平台。

五、答辩时的核心故事线
"传统静态分析工具在 CI/CD 流水线中表现优秀，但面对复杂业务逻辑、绕过式过滤、跨函数污点等场景时力不从心。本系统利用本地部署的大语言模型，通过RAG 知识库增强和语义级代码理解，在保证检出率的同时，生成可执行的修复代码和自然语言解释，将人工审计时间从 30 分钟缩短到 5 分钟。实验表明，在典型漏洞上 LLM 不弱于传统工具，在难样本上显著优于传统工具，证明了 LLM 在代码安全审计中的差异化价值。"
