# 已完成实验评估与下一步修复计划

> 用途：把当前已完成实验（exp_01 / exp_02 / exp_03）评估出的关键问题与修复建议传递给 AI，让其据此制定具体执行方案。原文件中已完成或已在其他文档（如规划.md / TODO.md）中覆盖的内容已删除。

## 一、当前关键结论

已完成的三阶段实验在工程实现上较为规范，但实验设计存在若干严重问题，若不及时修正，会导致 100% 召回率、0 误报等关键数据在论文/答辩中可信度不足。

## 二、八大修复建议（按优先级排序）

### P0-1：样本答案泄露，破坏盲测原则

**问题描述**

所有 14 个样本都在代码注释中写明了漏洞类型与存在性，例如：

```python
# 样本: 命令注入 - Python subprocess shell=True
# 期望: 检测到命令注入（用户输入拼接到 shell 命令中且 shell=True）
```

模型只要读到注释就能"作弊"，无需做真正的语义分析。这会直接动摇 exp_01 的 100% 召回率、exp_02 的 LLM vs 传统工具对比、以及 exp_03 的 RAG 不引入误报这三个核心结论。

**修复步骤**

1. 遍历 `experiments/exp_01_basic_scan/samples/` 下所有 `.py/.php/.js/.java` 文件
2. 删除所有形如 `# 期望: xxx`、`# 漏洞: xxx`、`# 安全: xxx`、`# 样本: xxx` 的注释
3. 保留必要的功能说明性注释（若有），但避免透露安全结论
4. 更新 `samples/manifest.json`，确保 ground truth 只存在于 `expected_present` / `expected_vulnerability` 字段中
5. 重跑 exp_01 / exp_02 / exp_03，获取无作弊的基线数据

**验收标准**

- 所有样本源码中不再包含任何可直接泄露漏洞答案的注释
- manifest.json 仍能正确标注期望结果
- 三个实验在无注释样本上重新运行并生成新的 results.json

**相关文件**

- `experiments/exp_01_basic_scan/samples/*`
- `experiments/exp_01_basic_scan/samples/manifest.json`

### P0-2：样本量不足且难度过低

**问题描述**

- 仅 14 个样本，漏洞样本 12 个、安全样本仅 2 个
- 安全样本 2 个导致误报率只能取 0/50/100% 三档，无统计意义
- 每类漏洞仅 2 个样本，类型准确率 12/12 的偶然性很高
- 所有样本都是 < 20 行的"教科书式"漏洞，模式极其明显

**修复步骤**

1. 将样本集扩充到 **30-50 段**，按以下比例设计：
   - 典型漏洞：占 50-60%（作为基线对照）
   - 安全对照：至少 6-8 段（覆盖 SQL 参数化、subprocess 列表、HTML 转义、路径校验等）
   - 难样本：占 20-30%，包括：
     - 绕过式过滤：`replace("'","")`、`strip()` 不完全过滤
     - 跨文件污点流：source 与 sink 不在同一文件
     - 真实 CVE PoC 片段
     - OWASP WebGoat / DVWA 典型题目片段
     - 长文件（300-500 行）中隐藏漏洞
2. 为每个样本建立 8 列 ground truth 标注：样本 ID、路径、CWE、风险等级、是否漏洞、触发条件、Source→Sink 污染路径、修复思路
3. 将新样本统一放入 `experiments/exp_04_hard_samples/samples/` 目录，并编写 `manifest.json`

**验收标准**

- 样本总数 ≥ 30
- 安全样本 ≥ 6
- 难样本 ≥ 6（覆盖至少 3 种难样本类型）
- 所有样本均附带完整的 8 列 ground truth 标注

**相关文件**

- `experiments/exp_04_hard_samples/samples/`（新建）
- `docs/必须手动学习的地方.md`（参考其标注表格式）

4. 更新 `exp_01_report.md` 中关于 schema 的描述，删除"首跑时 schema 仅有 4 个字段"的说明，或改为"历史版本已废弃"

- exp_01 的 raw_output 中模型实际输出的 JSON 块包含 7 字段
- `parsed_verdict` 中不再有 `null` 字段

### P1-4：每个样本仅跑一次，缺少重复性与置信区间

**问题描述**

temperature=0.1 并不等于确定性输出，Gemma 每次推理仍有随机性。当前每个样本只跑一次，无法区分"100% 准确"与"恰好这次对了"。

**修复步骤**

1. 为 `run_experiment.py` 和 `run_rag_experiment.py` 增加 `--repeat N` 参数（默认 N=3）
2. 对每个样本连续调用模型 N 次，记录每次的判定结果与耗时
3. 修改 `compute_detection_metrics()` 或新增函数，计算：
   - 每样本 N 次中判定为漏洞的比例
   - 按"多数表决"作为最终判定的整体指标
   - 总体准确率的 95% 置信区间（如 Wilson score interval）
4. 在报告中给出"均值 ± 标准差"的耗时，而非单点值

**验收标准**

- 每个样本至少跑 3 次，results.json 中保留每次原始输出
- 报告包含准确率 95% 置信区间
- 报告包含耗时的均值、标准差、中位数

**相关文件**

- `experiments/exp_01_basic_scan/run_experiment.py`
- `experiments/exp_03_rag_knowledge/run_rag_experiment.py`
- `experiments/utils.py`

### P1-5：RAG 缺少消融对照实验

**问题描述**

exp_03 只对比了"纯 LLM" vs "RAG+LLM"，没有排除"prompt 变长"或"随机注入文本"的干扰因素，无法严格证明 RAG 提升来自知识相关性。

**修复步骤**

1. 在 exp_03 中新增至少 3 组对照：
   - **A 组**：RAG+LLM（当前实现）
   - **B 组**：纯 LLM（无 RAG）
   - **C 组**：随机知识注入（从知识库中随机抽取 Top-K 条，与样本无关）
   - **D 组**：等长无关文本注入（注入与漏洞无关但长度相近的说明文字）
2. 在难样本集上运行上述 4 组实验，对比：
   - 检出率 / 误报率
   - 漏洞类型分类准确率
   - 输出结构质量（是否使用 Source/Sink 框架）
3. 只有当 A 组显著优于 B/C/D 组时，才能论证"RAG 有用"；否则只能说明"注入文本有用"或"无差异"

**验收标准**

- 4 组实验在同批难样本上完成
- 每组实验都有独立的 results.json
- 报告中包含对照实验表格与显著性结论

**相关文件**

- `experiments/exp_03_rag_knowledge/run_rag_experiment.py`
- `experiments/exp_03_rag_knowledge/exp_03_report.md`

### P2-6：Bandit 误报判定口径过宽

**问题描述**

exp_02 将 Bandit 所有 findings（包括 `B404 import_subprocess` 这种 import 级别的信息性告警）都计为"有漏洞"，导致 safe_02 被记为 3 条误报。这种口径对传统工具有失公平，容易被质疑。

**修复步骤**

1. 修改 `run_baseline.py` 中的 `run_bandit()`，解析每条 finding 的 `severity` 和 `confidence`
2. 默认只将 `severity` 为 LOW 及以上且 `confidence` 为 MEDIUM/HIGH 的告警计为漏洞
3. 提供 `--bandit-min-severity` 参数（如 `LOW` / `MEDIUM` / `HIGH`），允许调整口径
4. 在报告中明确说明所使用的判定口径，并对比不同口径下的结果

**验收标准**

- `B404 import_subprocess` 等信息性/LOW severity 告警不再直接等同于"漏洞"
- 报告中给出至少两种口径（严格 / 宽松）下的对比
- exp_02 结论应基于严格口径

**相关文件**

- `experiments/exp_02_baseline_tools/run_baseline.py`
- `experiments/exp_02_baseline_tools/exp_02_report.md`

### P2-7：耗时统计未处理异常值

**问题描述**

exp_01 中 safe_02 耗时 110.7s，约为平均值的 2 倍，但报告仍直接将其计入平均值，导致 56.7s/样本的指标被异常值拉高。

**修复步骤**

1. 在报告中同时给出：
   - 平均耗时
   - 中位数耗时
   - 标准差
   - 最大 / 最小值
2. 对 110.7s 这类异常值给出说明：是模型对安全样本过度分析导致，还是网络/加载抖动
3. 若确认是模型行为而非外部抖动，可在正文中保留，但论文核心耗时结论建议采用**中位数**或剔除异常值后的均值

**验收标准**

- exp_01 / exp_03 报告均包含中位数耗时
- 异常值有单独说明
- 论文中引用的"典型耗时"使用更稳健的统计量

**相关文件**

- `experiments/exp_01_basic_scan/exp_01_report.md`
- `experiments/exp_03_rag_knowledge/exp_03_report.md`

### P2-8：exp_03 未对比不同 Top-K 值

**问题描述**

RAG 固定 Top-3，未验证 Top-1 / Top-3 / Top-5 / Top-10 对准确率、耗时、可解释性的影响，缺少调参依据。

**修复步骤**

1. 使用 exp_03 的 `run_rag_experiment.py --top-k K`，分别跑 K=1,3,5,10
2. 在同批难样本上记录：
   - 检出率 / 误报率
   - Top-1 / Top-K 类型命中率
   - 平均检索 distance
   - 平均耗时
3. 绘制或表格展示 K 与指标的关系，给出推荐 K 值

**验收标准**

- 至少完成 K=1,3,5 三组对比
- 报告中包含 Top-K 对比表
- 明确推荐后续系统使用的 K 值及理由
