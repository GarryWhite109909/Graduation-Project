**`safe_03_subprocess_list.py`** [raw_output L249](file:///home/zane/文档/code/毕业设计/experiments/exp_06_finetune/results/exp_06_eval.finetuned_custom.20260710_030533.json#L249)

#### B. 概念混淆 + 幻觉（2 个，最诡异）—— SSTI 镜像错误

**`typical_23_ssti.py` (FN)** [L857](file:///home/zane/文档/code/毕业设计/experiments/exp_06_finetune/results/exp_06_eval.finetuned_custom.20260710_030533.json#L857)

> 代码：`env.from_string(f"Hello {name}")` —— 用户输入拼进模板字符串
> 模型说："template_str 是直接拼接的字符串，并没有经过 Jinja2 的模板引擎渲染，因此不会发生 SSTI"

**完全反了！** `from_string` 就是 Jinja2 的渲染方法，模型却说"没经过渲染"。

**`safe_15_ssti_escape.py` (FP)** [L1353](file:///home/zane/文档/code/毕业设计/experiments/exp_06_finetune/results/exp_06_eval.finetuned_custom.20260710_030533.json#L1353)

> 代码：`env.from_string("<h1>Hello {{ name }}</h1>")` —— 固定模板 + autoescape
> 模型说："Jinja2 模板引擎存在 SSTI 风险"

**镜像错误**：typical_23 把危险的 `from_string`（用户控制模板）当成安全，safe_15 把安全的 `from_string`（固定模板）当成危险。模型不理解 **SSTI 的关键在于"谁控制模板内容"**，而是把 `from_string` 本身当成 SSTI 的充分条件。还把 CWE 标成"CWE-89 SSTI"（CWE-89 是 SQL 注入）。

**根因**：对 Jinja2 API 语义的概念性误解，不是偏见问题。

#### C. 推理结论断裂（1 个 FN）—— 训练数据误导的迹象

**`typical_19_weak_random.py`** [L793](file:///home/zane/文档/code/毕业设计/experiments/exp_06_finetune/results/exp_06_eval.finetuned_custom.20260710_030533.json#L793)

**代码明明用的是 `random`，模型却说用了 `secrets`！** 推理完全正确，结论却反转，还捏造了事实来配合"安全"的结论。

这个最值得关注——**模型可能从训练数据里学到了"安全样本→输出 none"的模式**，导致即使推理出问题，也强行输出安全。这是被训练数据误导的迹象。

## 三、顽固旧知识能从模型底层优化掉吗？

**核心问题**：shell=True/subprocess 偏见（3 个 FP）是预训练先验，LoRA r=8 的 0.26% 参数撼动不了。

### 方案 A：DPO（Direct Preference Optimization）— 最直接

BiasDPO（2024, arXiv:2407.13928）已证明 DPO 能有效减少 LLM 偏见。

**原理**：DPO 的损失函数直接最大化 chosen（正确判断）的概率、最小化 rejected（错误判断）的概率。比 SFT 更直接地"惩罚"偏见。

**数据格式**（偏好对）：
```
prompt: <safe_03 的代码>
chosen: "列表 subprocess + shell=False → 安全"（正确推理+正确结论）
rejected: "shell=True → 命令注入"（条件反射+幻觉）
```

**实现**：TRL 库的 `DPOTrainer`，在 SFT 模型基础上做 DPO。
**成本**：需要构造 20-30 条偏好对，训练 1-2 epoch。
**风险**：DPO 有"likelihood displacement"问题（chosen 和 rejected 概率同时下降），需控制学习率。

### 方案 B：CCoT（Contrastive Chain-of-Thought）— 适合小模型

CCoT（2025, SOLAV Journal）在 Phi-2（2.7B）上用 LoRA 实现，准确率提升 12.5%。

**原理**：训练数据里不仅有正确推理，还有**明确标注的错误推理路径**，教模型识别"条件反射"是错误的。

**数据格式**：
```
### Incorrect Reasoning Path:
shell=True 所以不安全，host 可以注入 ; rm -rf / ...
### Correct Reasoning Path:
代码是 ["ping", "-c", "1", host] 列表形式，shell=False，; 不会经过 shell 解释...
```

**实现**：修改训练数据，加入对比负例。
**成本**：需手写 20+ 条对比样本。
**优势**：不需要额外训练框架，和现有 SFT 流程兼容。

### 方案 C：知识编辑（ROME/MEMIT）— 不推荐

理论上可以编辑模型权重删除特定知识，但：
- 实现复杂，需要定位知识存储的特定层
- 可能影响其他知识的完整性
- 不适合毕设范围

### 结论

| 方案 | 可行性 | 成本 | 效果预期 |
|---|---|---|---|
| **DPO** | ✅ 高（TRL 库支持） | 20-30 偏好对 + 2h 训练 | 直接惩罚偏见，最有效 |
| **CCoT** | ✅ 高（改数据即可） | 20+ 对比样本 | 教模型识别错误推理 |
| ROME/MEMIT | ❌ 低 | 实现复杂 | 不确定 |

**推荐**：先用 CCoT（改数据，零框架成本），如果效果不够再上 DPO。

## 四、幻觉优化方案

模型幻觉表现：safe_03 捏造攻击 payload、noise_06 捏造用户输入、typical_19 捏造"已用 secrets 模块"。

### 方法 1：CoT 结论一致性约束（零成本，立即生效）

在 SYSTEM_PROMPT_LITE 加一条约束：

> **结论一致性校验**：JSON 的 has_vulnerability 必须与 CoT 推理结论一致。若 CoT 识别出风险（如"弱随机""不安全"），JSON 不得标 false；若 CoT 未识别出风险，JSON 不得标 true。

**原理**：typical_19 的"推理对结论错"正是因为 CoT 和 JSON 之间没有一致性约束。
**成本**：改 prompt 一行，立即生效。
**局限**：只能约束"结论与 CoT 一致"，不能消除 CoT 本身的幻觉（如 safe_03 在 CoT 里就幻觉了攻击 payload）。

### 方法 2：Self-Verification（后处理校验）

生成 JSON 后，追加一轮校验：

> 上面的 CoT 识别了哪些风险？JSON 结论是否与之一致？若不一致，请修正 JSON。

**原理**：让模型自检 CoT→JSON 的一致性。
**成本**：增加一轮推理（~15s/样本）。
**效果**：能检测 typical_19 那种结论漂移。

### 方法 3：CCoT 对比训练（结合方案 B）

训练数据里加入"推理对但结论反转"的负例：
```
### Incorrect: CoT 识别 random.choices 是伪随机 → JSON 标 false → 捏造"已用 secrets"
### Correct: CoT 识别 random.choices 是伪随机 → JSON 标 true → 结论与推理一致
```

**原理**：教模型识别"结论漂移"是错误的。
**成本**：手写 10+ 条对比样本。

### 方法 4：评估脚本加严

evaluate.py 不只看 `has_vulnerability`，增加：
- vulnerability_type 是否匹配 expected_cwe
- CWE 编号是否正确
- source/sink 是否指向代码中真实存在的位置

**原理**：把"蒙对的 TP"降级为"部分正确"，暴露真实能力。
