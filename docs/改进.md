## 二、逐样本失败原因分析

### FP 类（安全样本误报）——3 个，baseline 和 finetuned 都错

**1. `safe_03_subprocess_list.py` → FP**

代码：`subprocess.run(["ping", "-c", "1", host], capture_output=True, timeout=5)` —— **列表形式，shell 默认 False，是安全的。**

模型输出却说："`host` 直接拼接到命令行中，没有进行任何转义或验证" —— **这是幻觉**。代码明明是列表 `["ping", "-c", "1", host]`，模型把它看成了字符串拼接。模型看到 `subprocess.run` + 用户输入 → 条件反射判定命令注入，连代码都没读对。

**根因：模型有"危险 API + 用户输入 = 漏洞"的启发式偏见，会扭曲事实来配合结论。**

**2. `safe_04_path_whitelist.py` → FP**

代码有 `ALLOWED_FILES` 白名单 + `abspath` + `startswith` 双重防御 —— 安全。

模型承认防御存在，但说"这个检查并不是严格的……可能会被绕过" —— **含糊其辞，编造了一个不存在的绕过**。它没有给出具体的攻击 payload，因为实际上绕不过。

**根因：同上。模型先定结论"有漏洞"，再倒推理由。**

**3. `safe_08_shlex.py` → FP**

代码用 `shlex.quote(host)` 转义后 `subprocess.run(f"whois {safe_host}", shell=True)`。`shlex.quote` 是有效防御，测试集标注为安全。

模型说"shell=True 使得命令字符串会被 shell 解释器解析，可能绕过转义" —— **过度保守**。`shlex.quote` 的设计目的就是在 shell=True 场景下防注入，模型不信任这个防御。

**根因：模型把 `shell=True` 当成"必定不安全"，不管实际防御是否有效。**

### FN 类（漏洞样本漏报）——5 个

**4. `hard_cve_02_python_log_injection.py` → FN（baseline 和 finetuned 都漏）**

代码：`logger.info(f"Login attempt from user: {username}")` —— 日志注入（CWE-117）。

模型的安全检查清单只查了：SQL注入、XSS、命令注入、路径穿越、硬编码凭证、反序列化。查完这 6 项没有，直接判安全。**日志注入根本不在它的检查范围里。** 模型还说"日志记录器会自动处理特殊字符"——错误。

**根因：漏洞分类法缺失。SYSTEM_PROMPT 的 `ANALYSIS_SCOPE` 只列了 6 类（SQL/XSS/命令/路径/凭证/反序列化+"等"），模型把"等"当成了"就这些"。训练数据的 CoT 模板也只演示这 6 类，强化了盲区。**

**5. `typical_17_md5_password.py` → FN（baseline 和 finetuned 都漏）**

代码：`hashlib.md5(password.encode()).hexdigest()` —— 弱密码学（CWE-327）。

模型检查完 6 项注入类，说"没有涉及到任何可能导致安全漏洞的操作"。**它根本没意识到 md5 用于密码哈希是弱算法。** 它的推理模板是"source→sink→防御"，没有"这个算法是否安全"这一步。

**根因：同上。弱密码学不在分类法里。**

**6. `typical_19_weak_random.py` → FN（baseline 和 finetuned 都漏）**

代码：`random.choices(string.ascii_letters + string.digits, k=16)` 生成重置 token —— 弱随机数（CWE-330）。

模型说"生成的令牌是随机的，并且长度固定，这有助于提高安全性" —— **完全反了**。`random` 模块不是密码学安全的，应该用 `secrets`。模型不懂这个区别。

**根因：同上。弱随机数不在分类法里。**

**7. `hard_longfile_01_hidden_sql.py` → FN（baseline）→ TP（finetuned！）**

长文件，9 个类全用参数化查询，只有 `StatsService.export_report` 用字符串拼接。

Baseline：扫了前几个类发现都用参数化查询，就停了，结论"所有数据库操作都使用了参数化查询" —— **注意力被稀释，没扫完。**

Finetuned：扫到了 `export_report` 的拼接，正确判定 TP。**finetune 在这里帮了忙**——训练数据里有长文件样本，教会了模型扫完全部方法。

**8. `hard_longfile_02_hidden_cmd.py` → FN（baseline）→ TP（finetuned！）**

同上模式，长文件里藏了 `subprocess.run(f"tar -cf {target}...", shell=True)`。Baseline 漏了，finetuned 抓到了。

## 三、三个根因总结

| 根因                                                     | 影响样本                                     | 是 finetune 能解决的吗                                       |
| :------------------------------------------------------- | :------------------------------------------- | :----------------------------------------------------------- |
| **分类法缺失**（日志注入/弱密码学/弱随机数不在检查范围） | hard_cve_02, typical_17, typical_19（3个FN） | ❌ 训练数据 CoT 模板只演示 6 类，反而强化了盲区               |
| **"危险API=漏洞"偏见**（扭曲事实配合结论）               | safe_03, safe_04, safe_08（3个FP）           | ❌ 训练数据漏洞样本第4步统一"无有效防御"，强化了"看到危险函数就报" |
| **长文件注意力稀释**                                     | hard_longfile_01/02（2个FN）                 | ✅ finetune 已解决（训练数据有长文件样本）                    |

**最关键的发现：你的 SYSTEM_PROMPT 的 `ANALYSIS_SCOPE` 只列了 6 类漏洞，这是 3 个 FN 的直接原因。** 改一行 prompt 就能修，不需要训练。

## 四、为什么训练后反而变弱

你的判断"数据集有误导"是对的，具体有三层原因：

**1. 训练数据的 CoT 质量低于 7B baseline 自身的推理质量**

7B baseline 是 Qwen2.5-Coder-7B，预训练见过海量代码安全语料，推理能力已经不错。但你的训练 CoT 是 7 套模板填空（"污染源：{source}。危险 sink：{sink}。防御检查：代码中无有效防御措施"），比模型自己的推理还粗糙。**你在用低质量 CoT 覆盖高质量推理——这叫"负蒸馏"。** 模型本来能做的细致分析，被训练成了模板复述。

**2. 训练数据强化了模型的两个偏见**

- 漏洞样本第4步统一"无有效防御措施" → 教模型"看到危险函数就不用看防御了，直接报" → FP 加重
- CoT 只演示 6 类注入 → 教模型"安全分析就查这 6 类" → FN 不改善

**3. 换皮增强（1866条）是纯噪声**

变量重命名 + 日志注入不改变语义，只增加模板记忆。1866 条低质样本把 7B 原本的知识冲淡了（catastrophic forgetting）。好消息是你已经写了去换皮的 `combine_and_augment.py`，但 e3_s42 大概率还是用的旧数据。

**简单说：7B 已经会跑了，你非要给它一本写得比它还差 的教材让它背，它当然越背越差。**

## 五、权重模型比 ollama 慢、占不满功耗的原因

这是正常现象，不是 bug：

| 维度     | ollama (llama.cpp)          | HF transformers generate()                  |
| :------- | :-------------------------- | :------------------------------------------ |
| 推理引擎 | C++ 高度优化内核            | Python + PyTorch，token 间有 GIL 和调度开销 |
| 量化     | GGUF 4bit/8bit，mmap 零拷贝 | fp16 全精度，权重占满显存                   |
| 注意力   | flash-attn 优化             | `sdpa` 在 ROCm 上未必最优                   |
| 批处理   | 连续批处理                  | batch_size=1，GPU 大量时间在等 CPU          |
| 功耗     | 持续满载                    | 单流推理下 GPU 利用率低，功耗上不去         |

**占不满功耗的核心原因**：`batch_size=1` 时，每生成一个 token，GPU 算一下就闲着等 Python 循环、KV cache 管理、采样逻辑。GPU 越强，等待占比越高，功耗越低。

**解决方案**（按推荐度）：

- **推理用 ollama 或 vLLM**：ollama 你已经在用，速度快 3-5 倍。如果要走权重推理，换 vLLM（支持连续批处理 + PagedAttention）。
- **`torch.compile(model)`**：编译图减少 Python 开销，提速 1.5-2x。
- **flash-attn**：ROCm 有 flash-attn 分支，比 sdpa 快。但安装复杂。
- **批量评估**：把 87 个样本批量送入而非逐条 `generate()`，GPU 利用率立即上来。

## 六、数据集怎么改

鉴于 7B baseline 已经 90.8%，你的策略要从"大量蒸馏"转向**"精准补盲"**：

### 第一步（零成本，立刻做）：改 prompt 修分类法盲区

3 个 FN（日志注入/弱密码学/弱随机数）的根因是 `ANALYSIS_SCOPE` 没列这些类。改 `prompts.py` 一行：

Python



```python
ANALYSIS_SCOPE = (
    "SQL 注入、跨站脚本（XSS）、命令注入、路径穿越、"
    "硬编码敏感信息（密钥/密码/Token）、不安全的反序列化、"
    "日志注入（CWE-117）、弱密码学（MD5/SHA1 哈希密码、CWE-327）、"
    "弱随机数（random 模块生成 token、CWE-330）、CSRF、"
    "SSTI、XXE、开放重定向、缺失认证/授权等"
)
```

这大概率直接修掉 3 个 FN，不需要训练。**先验证 prompt 修复后的 baseline，再决定要不要 finetune。**

### 第二步：如果要 finetune，数据集这样改

**原则：只训练 7B baseline 做不好的样本，CoT 质量必须高于 baseline 自身输出。**

| 改动             | 具体做法                                                     |
| :--------------- | :----------------------------------------------------------- |
| **砍量提质**     | 从 1866 条砍到 100-150 条，只保留 baseline 会错的类别。LIMA 证明 1k 高质 > 50k 低质 |
| **删掉换皮增强** | 彻底弃用 `augment_data.py` 和旧 `combine_and_augment.py`，只用新版 |
| **CoT 不用模板** | 漏洞样本第4步必须具体描述"为什么防御无效/不存在"，不能统一"无有效防御" |
| **补分类法盲区** | 新增日志注入、md5密码、弱随机数样本，CoT 演示"这个算法是否密码学安全"的推理 |
| **补 FP 对抗**   | 新增 safe_03/04/08 的同类型样本（列表subprocess、白名单+abspath、shlex.quote），CoT 必须显式推理"列表形式 shell=False 不触发 shell 解释" |

### 第三步：数据构造的具体方法

针对你 7B baseline 的 8 个错误，**手工写 8 类高质量的对抗样本**（每类 5-10 条），这是投入产出比最高的：

| 错误类型          | 要补的样本                                 | CoT 关键推理点                                               |
| :---------------- | :----------------------------------------- | :----------------------------------------------------------- |
| 日志注入 FN       | 用户输入进 `logger.info(f"...{x}")`        | "用户输入进日志，可注入换行符伪造日志条目（CWE-117）"        |
| 弱密码学 FN       | `hashlib.md5(pwd)` / `hashlib.sha1(pwd)`   | "MD5/SHA1 不适用于密码哈希，应用 bcrypt/argon2（CWE-327）"   |
