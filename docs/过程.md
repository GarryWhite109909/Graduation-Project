## 2026-06-29 难样本集扩充（P0-2）完成

### 背景

P0-1 修复后，虽然 14 个典型样本上仍保持 100% 准确率，但样本量小、难度低、安全对照样本不足，无法支撑毕设论文的统计意义和「LLM 优于传统工具」的核心论点。因此按 `docs/临时提示词_下一步计划.md` 中 P0-2 的验收标准，在 `experiments/exp_04_hard_samples/samples/` 下新建了扩展样本集。

### 本次工作内容

1. **样本集扩充至 42 段**
   - 典型漏洞：12 段（`typical_01~12`，覆盖 SQL 注入、XSS、命令注入、路径穿越、不安全反序列化、硬编码密钥、SSRF、open redirect、eval、YAML 反序列化等）
   - 安全对照：8 段（`safe_01~08`，覆盖参数化查询、HTML 转义、subprocess 列表、路径白名单、LIKE 参数化、CSP 头、输入校验、shlex 转义）
   - 难样本：16 段（`hard_bypass_01~04`、`hard_crossfile_01/02` 输入+sink、`hard_cve_01~04`、`hard_longfile_01/02`、`hard_owasp_01/02`）
   - 混淆噪音：6 段（`noise_01~06`，try-catch 掩盖、误导注释、硬编码拼接、注释掉危险代码、多层装饰器包装、shell=True 但输入硬编码）

2. **难样本覆盖 5 种类型**
   - 绕过式过滤（4）：SQL replace 双写绕过、命令 strip 绕过、XSS `<script>` 过滤绕过、路径正则绕过
   - 跨文件污点流（4）：`hard_crossfile_01/02` 输入层 + sink 层配对
   - 真实 CVE 片段（4）：CVE-2017-7494 Samba、CVE-2021-44228 Log4j、CVE-2025-4517 tarfile、CVE-2025-54381 BentoML SSRF
   - 长文件隐藏漏洞（2）：300+ 行隐藏 SQL 注入、260+ 行隐藏命令注入
   - OWASP / DVWA 风格（2）：任意文件上传、DVWA 经典 SQL 注入

3. **完整 ground truth 标注**
   - `manifest.json` 为每个样本提供 12 列标注：file / language / category / difficulty / expected_present / expected_vulnerability / expected_cwe / expected_risk_level / source / sink / taint_path / fix_idea
   - 所有 `manifest.json` 中的 file 都能在 `samples/` 目录找到，反之亦然

4. **统计意义提升**
   - 样本总数 42（≥ 30）
   - 安全对照样本 8（≥ 6），占比 38%（旧版 14.3%）
   - 难样本 16（≥ 6），覆盖 5 种类型（≥ 3）
   - 每类难样本至少 2 个实例

### 关键结论

- 扩展样本集已完成并满足 P0-2 全部验收标准。
- 当前处于「准盲测」状态：源码与 prompt 中均无答案泄露注释，但文件名（如 `typical_01_sql.py`、`hard_bypass_01_sql_replace.py`）仍可能向模型泄露漏洞类型。若后续要求严格盲测，可对文件名做中性化处理。

## 2026-06-29 主模型切换与硬件异响事件

### 背景

阶段四-2 的 P1-4 实验用 `gemma4:12b` 跑到第 23 个样本时，主机箱内突然出现明显异响。用户暂停实验并排查，怀疑是电源线或显卡供电线打到风扇，重新理线后异响仍在；进一步判断为 600W 电源在高负载下接近极限，风扇/电感产生异常噪音。决定下个月更换 1200W 电源后再继续大负载实验。

在实验暂停期间，对主模型进行了重新评估与切换。

### 本次工作内容

1. **模型选型测试**
   - 用户要求：纯文本、代码能力强、不浪费显存在多模态上
   - 候选：Qwen2.5-Coder 14B（代码专用 dense 模型）、DeepSeek-Coder-V2-Lite 16B（MoE，激活 2.4B）
   - 在 exp_01 前 5 个典型样本上做了快速对比测试

2. **三模型前 5 样本对比结果**

   | 模型 | 平均耗时 | 5 样本总耗时 | 有效判定 | 正确率 | 备注 |
   | --- | --- | --- | --- | --- | --- |
   | gemma4:12b | 40.62s | 203.11s | 5/5 | 5/5 | 原主模型，多模态 |
   | qwen2.5-coder:14b | **18.02s** | **90.08s** | 5/5 | 5/5 | JSON 格式规范 |
   | deepseek-coder-v2:16b | **9.00s** | **45.02s** | 3/5 | 3/5 | SQL 注入输出非法 JSON |

3. **主模型切换为 qwen2.5-coder:14b**
   - 修改 `src/llm_client.py` 默认模型
   - 修改 `experiments/exp_01_basic_scan/run_experiment.py` 默认 `--model`
   - 修改 `experiments/exp_03_rag_knowledge/run_rag_experiment.py` 默认 `--model`
   - 修改 `experiments/exp_03_rag_knowledge/knowledge_data/test_rag.py` 默认模型
   - 修改 `experiments/exp_04_hard_samples/run_experiment.py` 默认 `--model`
   - 修改 `experiments/exp_04_hard_samples/run_rag_experiment.py` 默认 `--model`
   - 修改 `experiments/exp_04_hard_samples/generate_report.py` 默认结果文件路径与模型说明
   - 更新 `README.md` 中模型说明、架构图、复现命令

4. **角色定位**
   - **主模型**：`qwen2.5-coder:14b` — 纯文本代码专用模型，速度是 gemma4:12b 的约 2.25 倍，准确率持平，JSON 输出稳定
   - **潜力模型**：`deepseek-coder-v2:16b` — 速度最快，但 `vulnerability_type` 字段输出非法 JSON（`"CWE-89", "SQL注入"`），需修复解析或 prompt 约束
   - **对照模型**：`gemma4:12b` / `gemma4:26b` / `gpt-oss:20b` — 用于横向对比

### 关键问题记录

1. **DeepSeek-Coder-V2-Lite 16B 的 JSON 格式问题**
   - 现象：模型把 `vulnerability_type` 输出成两个逗号分隔的字符串，缺少数组方括号
   - 示例：`"vulnerability_type": "CWE-89", "SQL注入",`
   - 影响：`src/schema.parse_verdict` 解析失败，两个 SQL 注入样本被判为 None
   - 后续方案：
     - 方案 A：在 prompt 中强制说明 `vulnerability_type` 必须是单个字符串
     - 方案 B：在 `parse_verdict` 中增加对这种畸形 JSON 的容错修复
     - 方案 C：作为多模型对比章节中的「格式遵循能力」分析点

2. **硬件异响与电源瓶颈**
   - 现象：运行 `gemma4:12b` 约 40 分钟后机箱内出现明显异响
   - 排查：重新绑扎风扇线后异响仍存在，判断为电源在高负载下产生
   - 根因：600W 电源在持续 100-150W CPU + 250W+ GPU 负载下接近极限，风扇/电感噪声异常
   - 决策：暂停长耗时实验，下个月更换 1200W 电源后再继续
   - 当前状态：
     - P1-4 纯 LLM 难样本测试：23/42 完成，19 个样本未完成
     - P1-5 RAG 消融：未开始
     - P2-8 Top-K 对比：未开始

3. **qwen2.5-coder:14b 的优势**
   - 纯文本模型，没有多模态参数浪费
   - 代码专用训练，对代码结构更敏感
   - 前 5 样本平均 18s，比 gemma4:12b 快 2.25 倍
   - 中文 prompt 理解好，JSON 输出格式规范
   - 32K 上下文，为后续长文件/CVE 样本提供更好的上下文支持

### 当前下一步
