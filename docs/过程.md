## 2026-06-30 DeepSeek 优化反思与主模型切换为 qwen2.5-coder:7b

### 背景

DeepSeek 安全样本优化专项完成后（`graduation_project/prompts.py` 增加安全模式白名单 + Few-shot 示例 + 硬编码凭证规则，`graduation_project/schema.py` 增加 `apply_safe_pattern_override` 兜底），exp_01 / exp_03 在 `deepseek-coder-v2:16b` 上均达到 recall=100%、FPR=0%、accuracy=100%。

但优化过程中暴露出两个问题：
1. **prompt 臃肿导致 deepseek 耗时翻倍**：SYSTEM_PROMPT 从 1663 字符膨胀到 5184 字符，deepseek 在 exp_01 平均耗时从 9.6s 涨到 26.8s（+180%），exp_03 从 14.8s 涨到 36.4s（+146%）。
2. **准确率提升并非来自 prompt 本身**：纯 Prompt 工程下 deepseek 安全样本误报率仍为 100%，最终 100% 准确率全靠 `safe_override` 后处理兜底。也就是说，臃肿 prompt 没有让 deepseek 学会正确判定，只是增加了推理成本。

用户因此提出：拉取参数更小的 qwen 模型与 deepseek 对比，看看是否更好；同时把主模型切到更优的基座。

### 本次工作内容

1. **拉取更小参数 qwen 模型**
   - `ollama pull qwen2.5-coder:7b`（约 4.5GB）
   - `ollama pull qwen2.5-coder:1.5b`（约 1GB）

2. **运行 exp_01 / exp_03 三模型对比实验**
   - deepseek-coder-v2:16b（完整 prompt + safe_override 兜底）
   - qwen2.5-coder:7b（完整 prompt，无兜底）
   - qwen2.5-coder:1.5b（完整 prompt，无兜底）

3. **尝试精简 prompt 验证必要性**
   - 完整版（5184 字符）
   - 超精简版（移除安全模式白名单）
   - 中等精简版（保留白名单，移除 Few-shot 示例与硬编码凭证规则）

4. **根据对比结果切换主模型**

## 2026-07-05 实验严谨性修复：样本答案泄露与 RAG 知识库去重

### 背景

在准备跑 exp_04 v3 多模型对比前，对样本集和 RAG 知识库做了一次全面审查，发现多处严重的严谨性问题，可能导致实验结论不可信。

### 发现的问题

1. **样本答案泄露（47 个文件）**：exp_04 样本集的 docstring 和注释直接告诉了模型漏洞类型、CWE 编号、漏洞位置。例如 `"""典型漏洞：SQL 注入（CWE-89）"""`、`# 漏洞：用字符串拼接构造模板`。模型可能靠读注释"作弊"而非分析代码语义。这让 exp_04 的所有实验结论失去意义——模型表现好可能只是因为会读注释。

2. **Ground truth 错误（1 个样本）**：`typical_29_integer_overflow.py` 在 Python 3 中整数无上限永远不会溢出，但 manifest 标注 `expected_present=true`。manifest 自承"考察模型语言特性理解"——故意设陷阱，模型诚实分析代码判 `false`（无漏洞）会被算作漏报，模型"作弊"按注释判 `true` 反而被算作正确。

3. **CVE 编号误导（2 个文件）**：`hard_cve_02_log4j_2021_44228.py` 文件名引用 Java Log4j JNDI 注入 CVE-2021-44228，实际代码是 Python 日志注入（CWE-117）；`hard_cve_04_ssrf_2025_54381.py` 文件名引用 BentoML SSRF CVE-2025-54381，实际代码是普通 urllib SSRF。文件名本身在误导模型。

4. **RAG 示例代码泄露（3 个条目）**：`sqli_safe_pattern` / `cmdi_safe_pattern` / `path_safe_pattern_strict` 三个 RAG 条目的示例代码与 safe_01/03/04 测试样本几乎一字不差（表名 users、字段 username/password、命令 `ping -c 1 host` 完全相同），构成"通过 RAG 泄露答案"。RAG 实验中模型检索到这些条目时，会直接"匹配"答案而非"理解"模式。

5. **RAG metadata 不一致（2 个条目）**：`pathtraversal_safe_pattern` 和 `idor_safe_pattern` 的 id 名为 `_safe_pattern` 但 metadata 缺少 `"safe_pattern": true` 标记，与同类 13 个条目不一致。

### 修复内容

1. **样本答案泄露修复**：删除 47 个文件中所有 `典型漏洞：`、`难样本：`、`安全对照：`、`# 漏洞：`、`// 漏洞：`、`# 安全：` 等泄露答案的注释。保留中性功能说明（如 `# 演示用 ORM 模型`），**漏洞代码本身完全保留不变**。

2. **Ground truth 修复**：将 `typical_29_integer_overflow.py` 改写为 Java 版本（Java `int` 是 32 位有符号整数，真有溢出），更新 manifest 中 language/source/sink/taint_path。

3. **CVE 编号误导修复**：重命名 `hard_cve_02_log4j_2021_44228.py` → `hard_cve_02_python_log_injection.py`，`hard_cve_04_ssrf_2025_54381.py` → `hard_cve_04_ssrf_urllib.py`，删除 manifest 中的 CVE 引用。

4. **RAG 示例代码去重**：修改 `sqli_safe_pattern`（users→accounts/email/status）、`cmdi_safe_pattern`（ping→nslookup）、`path_safe_pattern_strict`（/app/data→/srv/storage）的示例代码，让模型必须理解安全模式而非匹配代码。

5. **RAG metadata 一致性修复**：补齐 2 个条目的 `"safe_pattern": true` 标记，15 个 `_safe_pattern` 条目全部一致。

6. **ChromaDB 重建**：RAG 知识库内容修改后重建 ChromaDB（72 条知识，15 个 safe_pattern 标记一致）。

### 快速验证结果

| exp_03 全部 14 样本 | 14 | **100%** | 与修复前一致，证明 RAG 去重无负面影响——模型靠理解模式而非匹配代码 |

### 关键结论

2. **deepseek-coder-v2:16b 误报率最高**（44.4%）：存在幻觉式误报（把参数化查询说成"直接拼接 SQL"），印证了之前「DeepSeek 优化专项」的失败结论。
3. **qwen2.5-coder:7b 作为基座的合理性**：召回率 88.3% 略低，但体积小（4-5GB）、速度快（~7s/样本），作为安全专用模型微调基座最合适。
4. **实验结果整体可参考**：修正后数据已作为最终结论。

### 工程化增强：OllamaClient 支持 CPU+GPU 混合模式

审查过程中发现 26b 模型在 16k 上下文时 GPU 利用率低（用户反馈"功耗完全起不来，推理好慢"）。

**原因**：26b 模型 18.3 GiB > VRAM 16 GiB，Ollama 自动 25%/75% CPU/GPU 混合模式，但 GPU 在等待 CPU 完成那 25% 的层。

**修复**：`graduation_project/llm_client.py` 的 `generate()` 新增两个参数：
- `num_gpu`：GPU offload 层数（None=自动, 0=纯CPU, -1=全GPU, 正数N=前N层GPU其余CPU）
- `num_thread`：CPU 推理线程数（DDR5 高带宽场景可增大）

**用户偏好已记录**：大模型（20b+）默认使用 CPU+GPU 混合模式，配合 DDR5 高带宽。

### 踩坑记录

1. **真值错误的隐蔽性**：3 个"安全对照"样本的附带漏洞（硬编码凭证、反射型 XSS）非常隐蔽，编写样本时只关注了主防护点（PreparedStatement、CSRF token、bcrypt），忽略了响应输出的转义。这提醒我们安全对照样本需要全方位审查，不能只看主防护点。

2. **markdown 输出解析**：qwen2.5-coder:14b 偶尔输出 markdown 列表格式而非 JSON，这是模型行为的随机性。`parse_verdict` 需要兜底多种输出格式，包括 JSON 代码块、字段级正则、markdown 列表。

3. **子代理审查的价值**：本次审查启动了一个子代理做逐样本核查，发现了人工容易忽略的真值错误和解析错误。对于大规模实验结果，自动化审查 + 人工复核是必要的。

4. **Ollama 在 AMD GPU 上的混合模式**：Ollama 在 AMD Radeon GPU 上使用 ROCm 后端，当模型大于 VRAM 时会自动使用 CPU+GPU 混合模式。但默认的混合比例不一定最优，可以通过 `num_gpu` 参数手动调整。

## 2026-07-06：AST 代码切片实验（阶段四遗留收尾）

### 背景

v3 多模型对比结果审查时发现：qwen2.5-coder:7b 对 3 个 `hard_longfile_*` 样本虽二值判定全 True，但 longfile_01/03 的判定**理由错误**——模型把文件开头的 `DB_PATH="users.db"` / `app.secret_key` 误判为硬编码凭证（CWE-798），**漏掉了文件末尾隐藏的 SQL 注入与 SSTI**。这是长上下文注意力衰减的典型表现，也是阶段四遗留的待解决问题。

### 过程

#### 1. 工具选型与安装

- 选型：`tree-sitter` 0.20.4 + `tree-sitter-languages` 1.10.2（自带 grammar，离线可用）
- 踩坑：先尝试 `tree-sitter-language-pack` 1.12.2，但其需要从 GitHub 下载 grammar manifest，环境受限失败；回退到旧版 `tree-sitter-languages` + `tree-sitter 0.20.4`（API 兼容）
- 验证：Python / JavaScript / Java / PHP 四种语言 parser 自检通过

#### 2. 切片模块实现

新增 `graduation_project/code_slicer.py`：
- 输入：代码文本 + 语言 + 文件名
- 输出：`SliceResult`（含 `sliced` 标志 + `chunks` 列表）
- 策略：文件 < 150 行不切片；>= 150 行按顶层函数 / 类方法切分，每切片含文件级上下文头（imports / 全局常量 / 类骨架）+ 当前函数代码
- 阈值选择：默认 150 行（hard_longfile_03 181 行也被切片，验证 181 行已足以触发注意力衰减）

#### 3. run_experiment.py 改造

- 加 `--slice/--min-lines/--only/--filter` 四个参数
- 重构为 `run_sample_whole`（整文件模式）+ `run_sample_with_slice`（切片模式）两个函数
- 切片模式：每 chunk 单独送入 LLM，chunk 内 repeat 多数表决，**chunk 间 OR 合并**（保守策略宁误报不漏报）
- 统计指标修复：切片模式下用 sample 级别 `majority_verdict`（OR 合并结果），而非从 chunk runs 重新多数表决（会导致 longfile_01 因 5/20 chunk True 被判 False 的错误）

#### 4. 实验结果（repeat=3）

| 样本 | 整文件理由（v3 基线） | 切片后理由 | 改进 |
| --- | --- | --- | --- |
| longfile_01 | CWE-798 硬编码凭证（错） | **CWE-89 SQL注入**（3/3 一致） | ✓ 理由修正 |
| longfile_02 | CWE-78 命令注入（正确） | CWE-78 命令注入（2/3 一致） | — 已识别 |
| longfile_03 | CWE-798 硬编码凭证（错） | welcome chunk 3/3 判 False（漏 SSTI） | ✗ 未改进 |

- Sample 级别：TP=3, FN=0, 召回率 100%（OR 合并）
- Chunk 级别：126 chunks, TP=45, FN=81, chunk 召回率 35.7%
- 误报影响：仅 3 个 longfile >= 150 行会被切片，其余 84 个样本不受影响

### 结论

1. **AST 切片对长文件隐藏的 SQL 注入有显著改进**：模型聚焦 `export_report` 函数后，3/3 一致判定 CWE-89 SQL注入，不再被文件开头的 `DB_PATH="users.db"` 干扰。这是 AST 切片的核心价值——**让模型注意力聚焦到漏洞函数**。
2. **AST 切片对需要领域知识的漏洞（如 SSTI）效果有限**：模型在 `welcome` chunk 上 3/3 判 False，解释"模板引擎默认自动转义"。这是模型对 SSTI 的理解问题（f-string 拼接模板字符串本身是 SSTI，与自动转义无关），**需要 RAG 知识辅助**，非切片能解决。
3. **AST 切片 + RAG 是互补方案**：切片解决"注意力衰减"，RAG 解决"知识缺失"，两者结合可进一步提升长文件难样本的检出质量。
4. **保守 OR 合并策略的代价**：sample 级别召回率 100% 但 chunk 级别仅 35.7%，意味着大量非漏洞函数被误判 True，依赖 OR 合并"碰巧"覆盖到真正的漏洞函数。后续可探索更精细的合并策略。

- `experiments/exp_04_hard_samples/results/results.slice-longfile.repeat1.qwen2.5-coder-7b.json`：repeat=1 结果
- `experiments/exp_04_hard_samples/results/results.slice-longfile.repeat3.qwen2.5-coder-7b.json`：repeat=3 结果

- `requirements.txt`：加 tree-sitter 0.20.4 + tree-sitter-languages 1.10.2

### 踩坑记录

1. **tree-sitter 版本兼容性**：`tree-sitter-language-pack` 1.12.2 需要联网下载 grammar，`tree-sitter-languages` 1.10.2 需要 `tree-sitter 0.20.x`（不能用 0.26+）。最终选 `tree-sitter 0.20.4` + `tree-sitter-languages 1.10.2`，离线可用且 API 稳定。

3. **AST 切片的边界**：切片只能解决"模型看到漏洞函数"的问题，不能解决"模型不认识漏洞模式"的问题。longfile_03 的 SSTI 就是后者——模型聚焦到 `welcome` 函数后仍判 False，因为模型认为 Jinja2 自动转义会防护（实际上 f-string 拼接模板字符串本身是 SSTI）。这提示后续应探索切片 + RAG 组合方案。

## 2026-07-06：Prompt 工程消融实验（exp_05）

### 背景

阶段四遗留的 AST 切片完成后，转向 prompt 工程消融对比。核心问题：当前 SYSTEM_PROMPT 是经验性拼装的（白名单 + 硬编码规则 + 6 条要求 + schema），各部分的独立贡献不明。需要系统化对比 5 种 Prompt 策略，回答：白名单是否"作弊"？Few-shot 是否带来增益？CoT 是否提升难样本召回？组合策略是否最优？

### 过程

1. **5 个变体设计与实现**（`graduation_project/prompts.py`）：
   - `zero_shot`：当前完整版 SYSTEM_PROMPT（2966 字符），含白名单+硬编码规则+6 条要求+schema
   - `whitelist_only`：仅角色 + 白名单 + schema（2019 字符），去掉 6 条要求与硬编码规则
   - `few_shot`：zero_shot + 3 组示例（4259 字符），示例刻意与 manifest 样本不同避免答案泄露
   - `cot`：zero_shot + 5 步思维链要求（3348 字符），引导模型按 source→sink→防御→有效性→结论 分析
   - `combined`：zero_shot + CoT + Few-shot 三合一（4641 字符）

2. **实验脚本开发**（`experiments/exp_05_prompt_ablation/run_ablation.py`）：
   - 复用 exp_04 的 87 段样本，保证横向对比公平
   - 支持参数：`--variants` 选变体、`--repeat` 多次重复、`--resume` 断点续跑、`--filter` 按文件名筛选
   - 自动卸载模型（除非 `--keep-loaded`），跨文件样本自动注入对应 `_input.py` 上下文

3. **跑全部 5 变体 × 87 样本 = 435 次推理**（qwen2.5-coder:7b，温度 0.1）：
   - 第一轮 bug：`upsert_sample` 按 file 替换，导致前 4 个变体被 combined 覆盖，最终 json 只剩 combined 数据
   - 修复：本地实现 `upsert_sample_variant`，按 (variant, file) 联合 key upsert
   - 第二轮 `--resume` 跑前 4 个变体（combined 已有数据），脚本主循环只算 --variants 指定的变体 metrics，combined 的 metrics 又被覆盖
   - 第三次修复：写一次性脚本从 samples 重新计算所有 5 个变体的 metrics 写回 json

4. **按样本类别细分统计**：typical（36 全漏洞）/ hard（24 漏洞+3 安全）/ safe（18 全安全）/ noise（6 全安全），各类别召回与误报分别对比

### 结论

| 变体 | TP | TN | FP | FN | 召回率 | 误报率 | 准确率 | 平均耗时 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| zero_shot | 52 | 22 | 5 | 8 | 86.7% | 18.5% | 85.1% | 7.25s |
| whitelist_only | 44 | 23 | 4 | 16 | 73.3% | 14.8% | 77.0% | 6.67s |
| few_shot | 56 | 19 | 8 | 4 | 93.3% | 29.6% | 86.2% | 7.20s |
| **cot** | **57** | **21** | 6 | **3** | **95.0%** | 22.2% | **89.7%** | 8.61s |
| combined | 55 | 21 | 6 | 5 | 91.7% | 22.2% | 87.4% | 7.77s |

**核心发现**：

1. **CoT 是最优单一策略**：召回 95.0% / 准确率 89.7%，优于 combined 87.4%——"prompt 越长越好"是误区。CoT 的 5 步分析显式引导污点分析，对难样本召回 95.8%（最高）。
2. **白名单是"压误报但损召回"策略**：whitelist_only 误报最低（14.8%）但召回也最低（73.3%）。这反驳了"白名单作弊到直接给答案"的质疑——光有白名单反而让模型对真漏洞过于宽容，必须配合"不要遗漏明显漏洞"等召回引导要求。
3. **Few-shot 是双刃剑**：召回 +6.6pp 但误报 +11.1pp，净收益不明显。示例让模型形成"看到用户输入就判漏洞"的偏见，导致跨文件 input 误报暴涨。
4. **组合策略不是最优**：combined 准确率 < cot，prompt 过长导致注意力分散，CoT 步骤与 few-shot 示例存在注意力竞争。
5. **prompt 工程有天花板**：cot 仍有 1 个难样本（hard_bypass_04_path_regex.py）漏报，4/5 变体都漏。论证了后续 LoRA/QLoRA 微调的必要性。
6. **跨文件污点是 prompt 工程盲区**：crossfile_input 在多数变体上误报，需 RAG 或代码图谱解决。

### 踩坑记录

1. **upsert_sample 通用函数的隐患**：通用 `upsert_sample` 按 file 字段替换，适用于"一样本一记录"场景。但消融实验是"一样本 N 变体 N 记录"，按 file 替换会让前一个变体的记录被后一个变体覆盖。修复：本地实现 `upsert_sample_variant` 按 (variant, file) 联合 key。
2. **--resume 与 metrics_by_variant 的冲突**：`--resume` 跑指定变体时，主循环结束后只算 `--variants` 指定的变体 metrics，已存在 json 里的其他变体 metrics 被覆盖丢失。修复：写一次性脚本从 samples 重新计算所有变体 metrics。后续应改主脚本：metrics 计算应基于 samples 里实际存在的所有变体，而非 `--variants` 参数。
3. **CoT 在 safe 类样本上误报最高（22.2%）**：CoT 让模型"想太多"，对简单安全样本过度分析导致误判。这是 CoT 唯一短板，但总体准确率仍最高。
