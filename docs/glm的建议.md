### 🔥 必做（不做毕设就站不住）
1. 第二阶段对比实验 — 这是答辩时"为什么用 LLM 而不是 Bandit"的核心论据。同样 14 个样本跑 Bandit/Semgrep，做出 LLM vs 传统的对比表（检出/漏报/误报/耗时/解释质量）。 没有这个，"改进点"就站不住脚 。
2. 真实代码测试 — 现在的 14 个样本是"教科书漏洞"，太简单。从 CVE PoC 仓库、OWASP WebGoat 抽 10-20 段真实代码，看模型在"非典型写法"上的表现。这能直接暴露 LLM 的真实能力边界。
### 💡 创新点（最能体现"高级性"）
3. RAG 漏洞知识库 — 把 OWASP Top 10、CWE 条目、CVE 描述入库（Chroma 向量库），每次分析前先检索相关漏洞模式塞进 Prompt。 这是最容易做出"创新性"的一点 ，答辩时直接演示"有 RAG vs 无 RAG 的检出率对比"。
4. AST 代码切片 — 用 tree-sitter 把长文件切成函数/代码块再喂给 LLM，解决"长上下文注意力衰减"。这个能写一整章论文（"基于 AST 的代码切片策略对检测召回率的影响"）。
5. 多模型对比实验 — Gemma 4 26B vs Llama 3 vs Qwen2.5，不只跑一个。结论"本地 26B 已接近商业模型"或"X 类漏洞 Gemma 强、Y 类 Llama 强"都有论文价值。
### ⭐ 加分项（做了惊艳）
6. 污点流分析 — Source→Sink 追踪，不只单点检测。可以用 LLM 做"污点传播推断"（这是传统工具的弱项）。
7. 修复建议质量评分 — 不只看检出率，对 LLM 生成的修复代码做"是否能通过测试/是否能编译"的量化评估。这个指标很少有人做。
8. Prompt 工程对比 — 零样本 vs Few-shot vs 思维链，在难样本上对比。可以单独写一节。
9. 工程化系统 — Web 界面 + 报告导出（PDF/Markdown）+ 批量扫描。让"系统"看起来像系统，不是脚本。
