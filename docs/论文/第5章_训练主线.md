## 5.4 Phase 4:Prompt Distillation 修复 CPT 副作用

### 5.4.1 动机与设计思路

Phase 3 的 KnItLM CPT 路线在严格召回率(+23pp)、误报率(-7.7pp)、CWE 错标数(-16)三项核心指标上均取得突破,但错题分析(详见 5.3.5 节)揭示了一个不可忽视的副作用:模型在 4 个 Phase 1 答对、Phase 3 答错的样本上出现了"参数化查询幻觉"——将使用 `?` 占位符的参数化查询误判为字符串拼接 SQL 注入。这一副作用源于 CPT 语料中漏洞样本占比偏高,模型在习得"识别拼接型 SQL 注入"模式的同时,过度泛化到所有含字符串拼接语法的 SQL 上下文,模糊了"拼接构成漏洞"与"参数化查询是安全模式"的边界。

直接降低 CPT 语料中漏洞样本占比会削弱 Phase 3 的召回率优势,因此本研究采用 Prompt Distillation 路径修复这一副作用,而非调整 CPT 语料配比。基本思路是引入一个具备更强代码语义理解能力的 teacher 模型,通过 KL 散度约束 student 模型在 answer token 上的 logits 分布向 teacher 靠拢,从而在不损失 Phase 3 注入知识的前提下,校准 student 的推理分布、修复过度泛化。

Phase 4 的设计有以下三个关键决策。

**决策 1:Teacher 选择 Qwen3-Coder-30B(MoE)。** 该模型为 30B 参数规模的 Mixture-of-Experts 模型,激活参数约 7B,推理速度与 dense 7B 相当,但在代码理解任务上具备接近 dense 30B 的能力。由于 teacher 仅用于预计算 logits(每个样本的 top-K token 概率分布),不参与训练反向传播,可以部署在 Ollama 后端通过 CPU+GPU 混合模式运行,占用 0 GPU 显存,与 student 训练共享同一张 16 GB 显卡。这一设计避免了 teacher 与 student 同时驻留显存导致的 OOM 问题。

**决策 2:Student 基于 Phase 3 KnItLM 合并模型(路线 B)。** Phase 4 student 有两种选择:路线 A 是基于原始 Qwen2.5-Coder-7B-Instruct,独立验证 Prompt Distillation 的效果;路线 B 是基于 Phase 3 KnItLM 合并模型,叠加"知识注入 + 蒸馏校准"两阶段训练。本研究采用路线 B 作为主路径,理由是 Phase 3 已注入的领域知识应被保留并校准,而非从零开始蒸馏。路线 A 作为 ablation 对照在 5.4.7 节给出。

**决策 3:Loss 函数采用 (1-α)·CE + α·KL 形式,α=0.5。** CE 项保证 student 不偏离 ground-truth 标签,KL 项约束 student 分布向 teacher 靠拢。α=0.5 表示两项等权,既不放弃监督信号也不过度依赖 teacher 软标签。温度系数 T=2.0 按 Prompt Distillation 原论文(TMLR 2025)推荐值设置,使 teacher 的软标签分布更平滑,暴露更多 dark knowledge。

### 5.4.2 Teacher 模型与 logits 预计算

Prompt Distillation 的训练效率瓶颈在于 teacher logits 的获取。若每个训练 step 都调用一次 teacher 前向传播,30B MoE 模型即使激活参数仅 7B,也会使单步训练时间从约 2s 暴涨到 8s 以上,且 teacher 与 student 共享显存极易 OOM。本研究采用预计算策略:训练前一次性调用 teacher 对全部训练样本(823 条 CoT 蒸馏样本)做前向传播,保存每个样本 answer 部分每个 token 位置的 top-K(K=20)token index 与 softmax 概率,训练时直接从磁盘加载,与 student logits 对齐计算 KL 损失。预计算阶段 teacher 部署在 Ollama 后端,CPU+GPU 混合模式运行,823 样本约耗时 4 小时;训练阶段 teacher 不再参与,单步训练时间回归到约 2s。

Top-K 近似的合理性在于:teacher 分布的绝大多数概率质量集中在前 20 个 token(实测覆盖率 > 95%),尾部长尾分布对 KL 损失贡献极小。K=20 也是 Ollama 后端返回 top-K 的硬上限。
